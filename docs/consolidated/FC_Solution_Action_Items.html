🏗️ Architecture

Core architecture decisions — what stays, what changes, what's overkill
A1. Multi-Tenant RLS + Shared Schema
Keep

All 3 AIs validated this. RLS with shared schema is correct for this stage. Schema-per-tenant would slow development, complicate migrations, and add operational drag.

What Exists:
RLS policies on all tenant tables via migration 001. Tenant context middleware sets app.current_tenant before every query. Record history triggers on all tables.
3-Eye Consensus: All 3 agree — keep. No changes needed.
A2. Transactional Outbox Pattern
Keep

Domain write + outbox insert + saga update + audit log in single transaction. Zero message loss guarantee. GPT called this "appropriate for a compliance-heavy integration product."

What Exists:
outbox-poller.ts — polls outbox_events table, dispatches to BullMQ, marks processed. 4-write transaction pattern in order-service.ts.
3-Eye Consensus: All 3 validated. Keep as-is.
A3. Idempotency Keys
Keep

X-Idempotency-Key header → Redis check → DB check → process. Critical for EDI where duplicate submissions cause chargebacks.

What Exists:
middleware/idempotency.ts — full implementation with Redis + DB fallback.
A4. Audit Trail with Hash Chain
Keep

SHA-256 hash-chain audit log. Non-negotiable for compliance-heavy product. GPT explicitly called this out as appropriate.

What Exists:
audit_trail table with hash chain verification. Immutable append-only.
A5. Move ALL Intelligence to Cloud
RedesignCritical

All transformation, validation, EDI generation, mapping, and orchestration must live in the cloud API. Bridge Agent becomes read-push only. This is the single most important architectural shift.

What Exists:
Current design has some transformation logic planned for Bridge. EDI adapters (x12-adapter.ts, cxml-adapter.ts, json-adapter.ts) are already in the cloud API.
What Changes:
Ensure no canonical transformation, validation rules, or document generation runs on the Bridge. Bridge sends raw ERP data; cloud does everything else. Remove any Bridge-side transformation plans from architecture docs.
3-Eye Consensus: All 3 agree — keep Bridge dumb. GPT: "Leave only extraction, buffering, and local retry." Gemini: "Keep the Agent as dumb as possible (Read/Push only)."

D2. Mapping Registry Table
NewCritical

All 3 AIs identified this as the real moat. A versioned, reusable library of buyer-specific mapping templates. When you map Tally→H&M once, every H&M supplier becomes a 5-minute setup.

New Table: mapping_templates
id, buyer_id, buyer_name, document_type (PO/ASN/Invoice), edi_standard (X12/EDIFACT/cXML), version, field_rules (JSONB — source→target mappings), validation_rules (JSONB), transform_rules (JSONB — unit conversions, date formats), created_from_factory_id, usage_count, success_rate, is_golden (verified by FC team), created_at, updated_at.

3-Eye Consensus: All 3 agree — this IS the moat. GPT: "proprietary onboarding data: field dictionaries, buyer schema libraries, validation rule packs." Gemini: "Once you map Tally to H&M once, every other H&M supplier becomes a 5-minute setup."
D3. Chargeback Tracking Tables
New

Chargeback prevention is the strongest sales pitch. Shadow Mode needs to calculate "FC saved you ₹X" — that requires tracking chargeback events and their root causes.

New Table: shadow_mode_reports
factory_id, report_period, total_documents_processed, errors_caught, chargebacks_prevented, estimated_savings, accuracy_score. This feeds the sales conversion report.
D4. Document Metering Table (for per-document billing)
New

Gemini recommended ₹5k base + per-document fee instead of flat pricing. This requires document-level metering from day one.

New Table: document_meter
factory_id, document_type, direction (INBOUND/OUTBOUND), billing_period, document_count, billable_count, unit_rate, total_amount. Increment on every document processed.
D5. Simplify Connection/Partner Schema
Redesign

Current schema has both connections and partner_systems tables with overlapping concerns. Simplify to a single buyer_connections table focused on what Phase 1 needs.

What Exists:
migration 004 — partner_systems with protocol configs, SLA settings, certificate storage.
What Changes:
Keep partner_systems but defer certificate management, SLA monitoring, and protocol-level config until a real buyer integration demands it. Add mapping_template_id FK to link buyer connections to the mapping registry.

🔄 Saga & Workflow Engine

The biggest redesign — all 3 AIs flagged the 15-state saga as overkill
S1. Simplify 15-State Saga to 5-State Workflow
RedesignCritical

This is the highest-consensus action item across all 3 AIs. The current 15-state saga coordinator is enterprise-grade machinery for a product that hasn't processed its first real transaction.

What Exists (15 States):
RECEIVED → VALIDATED → MAPPED → QUEUED → SENT → DELIVERED → ACKNOWLEDGED → ACCEPTED → REJECTED → RETRYING → FAILED → CANCELLED → COMPENSATING → COMPENSATED → DEAD_LETTER. Full saga-service.ts with SLA monitoring, lock recovery, deadline tracking.
Phase 1 Replacement (5 States):
RECEIVED → MAPPED → VALIDATED → SENT → ACKNOWLEDGED

Plus two terminal states: FAILED, CANCELLED.

What gets removed: SLA monitoring, lock recovery, compensation sagas, dead letter handling, deadline tracking. These are all real needs — but for 10+ customers, not for customer #1.

What replaces it: Simple state machine with BullMQ retry on failure. If a document fails 3 times, it goes to a human exception queue. That's it.

Keep the saga_service.ts file but gut the internals. The state machine transition logic stays (validated transitions only), but remove the distributed coordination overhead.
Files to change: apps/api/src/services/saga-service.ts packages/shared/src/types/enums.ts (SagaState enum)
3-Eye Consensus: All 3 agree. GPT: "Simplify the saga into a narrower workflow engine." Gemini: "Saga Coordinators and 15 states are overkill. A retry queue is enough." Claude: "Replace with 5-state workflow."
S2. Exception Queue (Human-in-the-Loop)
Redesign

Instead of auto-compensation sagas, build a simple exception queue where failed documents land for human review. The factory's export manager resolves them manually — this IS the workflow for the first 5 customers.

New Implementation:
exception_queue table: document_id, factory_id, error_code, error_detail, assigned_to, status (OPEN/IN_REVIEW/RESOLVED/ESCALATED), resolution_notes, created_at, resolved_at. BullMQ worker moves FAILED documents here after max retries. Portal shows this as a simple list with filters.

🌉 Bridge Agent

On-premises agent for factory Windows PC — make it dumber, more resilient
B1. Strip Bridge to Read-Push-Buffer Only
RedesignCritical

Bridge Agent responsibilities after redesign: (1) Read from Tally XML port 9000, (2) Buffer in local SQLite, (3) Push raw data to cloud via WebSocket/HTTPS. Nothing else.

What Exists:
apps/bridge/src/ — tally-connector.ts (XML extraction), adaptive-poller.ts (5s-5min interval), cloud-sync.ts (WebSocket tunnel), local-queue.ts (SQLite offline buffer), erp-orchestrator.ts (multi-ERP coordination).
What Changes:
Keep: tally-connector.ts, adaptive-poller.ts, local-queue.ts (SQLite buffer), cloud-sync.ts basic push.
Remove from Bridge scope: Any field mapping, validation, transformation, EDI generation. Remove zoho-connector.ts and sap-b1-connector.ts for Phase 1 — Tally only.
Add: Health-check reporting (port 9000 status, Tally version, TDL status, connectivity, disk space, last successful sync).
3-Eye Consensus: All 3 agree. Gemini: "Minimize the Windows Support Nightmare (updates, AV, registry locks)." GPT: "The biggest architectural risk is edge reliability and install friction."
B2. Tally TDL Push Events (Replace Pure Polling)
New

Gemini flagged a critical gap: Port 9000 polling alone isn't sufficient. Custom TDL scripts can push events on voucher save, giving near-real-time data without constant polling overhead.

New Implementation:
Build a minimal TDL (Tally Definition Language) script that fires an HTTP POST to the Bridge Agent's local endpoint whenever a voucher is saved/altered/deleted. Bridge Agent runs a tiny HTTP server on localhost to receive TDL push events. Fall back to polling when TDL isn't installed. This hybrid approach (TDL push + polling backup) is the most resilient.
Gemini unique insight: "Relying on adaptive polling without a custom TDL script to push events will lead to data latency and missed orders."
B3. Factory Environment Health Dashboard
New

Before any data flows, the agent must validate the factory environment. This is the "pre-flight check" that prevents failed installations.

Health Probes:
Port 9000 reachable (Y/N), Tally version detected, TDL installed (Y/N), Windows firewall status, antivirus interference check, internet connectivity to FC cloud, SQLite disk space, last successful Tally read timestamp, XML parser error count (last 24h). Surface all of this in the Portal as a "Factory Health" widget.
B4. Remove Multi-ERP Support for Phase 1
Defer

Zoho Books and SAP B1 connectors exist in code but are distractions. Tally is 95% of the target market. Build for one ERP first.

Defer:
zoho-connector.ts, sap-b1-connector.ts, erp-orchestrator.ts multi-ERP routing. Keep the files but feature-flag them OFF. Re-enable when a paying customer needs Zoho or SAP B1.

📡 EDI & Integration Layer

What to build, what to skip, and the pragmatic "manual upload" strategy
E1. Output Perfect Files First, Skip Direct API
Redesign

Gemini's sharpest pragmatic insight: Don't build direct Ariba/Coupa API integration yet. Generate perfect EDI/cXML files that factories can upload manually to buyer portals. This proves 90% of the value while bypassing 6 months of supplier onboarding red tape.

What Exists:
x12-adapter.ts (850/855/856/810 parse+generate), cxml-adapter.ts (PO/Confirm/Ship/Invoice), json-adapter.ts (REST webhook format). All have generate() methods.
What Changes:
Phase 1: Generate downloadable EDI/cXML files. Add file export endpoints (GET /api/documents/:id/download). Factory downloads and uploads to buyer portal manually.
Phase 2: Add direct API submission (Ariba PKI, Coupa OAuth, SPS Commerce SFTP) when a customer explicitly needs it.
Keep the adapter code — it's well-built. Just change the "delivery" step from API submission to file generation + download.
Gemini: "Direct Ariba/Coupa integration is a vanity metric. Start by outputting perfect EDI files." GPT: "Real timelines will be dominated by customer-side configuration, buyer-side onboarding, and exception handling, not coding."
E2. Defer OpenAS2 Integration
Defer

AS2 is the gold standard for EDI transport, but it requires buyer-side setup (certificates, trading partner config). Not needed when strategy is "generate files for manual upload."

Defer:
OpenAS2 Docker integration, certificate management, AS2 trading partner configuration. Re-enable when first customer needs automated EDI delivery over AS2.
E3. Build 3 Gold Standard Buyer Templates
New

Gemini's action item #4: Obtain/build specs for 3 representative buyers and create templates NOW.

Target Buyer Templates:
1. Walmart — X12 850 (PO), 855 (PO Ack), 856 (ASN), 810 (Invoice). Strictest requirements, highest chargeback risk.
2. H&M or similar EU buyer — EDIFACT ORDERS/DESADV/INVOIC. Tests EU format compliance.
3. Indian domestic buyer (Reliance/Tata) — Custom format or cXML via Ariba. Tests domestic use case.

Store as JSON in packages/shared/src/edi/templates/ and seed into mapping_templates table.
E4. Keep EDI Adapter Architecture
Keep

The pluggable adapter pattern (X12, cXML, JSON) is well-designed. Keep it — just add the "file download" output mode alongside future "API submit" mode.

What Exists:
x12-adapter.ts, cxml-adapter.ts, json-adapter.ts — all implementing parse() and generate() methods with proper envelope handling.

🏛️ GST-First Strategy (Gemini's Key Insight)

Enter through mandatory compliance, expand to voluntary EDI
G1. GST e-Invoice (NIC) Integration
NewCritical

Every Indian factory MUST generate e-Invoices. By capturing this data, FC gets 90% of what's needed for EDI 810 (Invoice) and 856 (ASN). This is the zero-friction entry point.

New Service: gst-service.ts
NIC API Integration:
— Auth: Client ID + Client Secret → auth token (valid 6 hrs)
— Encryption: RSA public key from NIC + AES-256 for payload
— Generate IRN: POST /eiapi/v1/Invoice with signed JSON
— Cancel IRN: POST /eiapi/v1/Invoice/Cancel (within 24hrs)
— IP Whitelist: Max 4 IPs (critical constraint — cloud API only, not Bridge)

e-Way Bill:
— Generate from e-Invoice data: POST /ewbapi/v1/ewayapi
— Auto-generate when invoice value > ₹50,000
— Track validity period (distance-based: 100km/day)

Data Flow: Tally invoice data → Bridge → Cloud → GST service generates e-Invoice → Same data feeds EDI document generation.
New files: apps/api/src/services/gst-service.ts apps/api/src/routes/gst.ts packages/shared/src/types/gst.ts
Gemini: "You cannot sell a Dispatch solution in India without this." GPT: "GST e-invoice/e-way bill not started is serious if you want to sit in live dispatch flows."
G2. GST-to-EDI Data Derivation Layer
New

Map GST e-Invoice fields to EDI document fields automatically. This is the bridge between mandatory compliance and buyer-specific EDI.

Field Mapping:
GST IRN → EDI Invoice Number, GSTIN → Supplier Tax ID, HSN Code → Product Classification, Place of Supply → Ship-To, e-Way Bill → ASN Transport Reference, Line items → EDI line items with UOM conversion. Build as a transformer in the mapping engine that auto-populates EDI fields from GST data, reducing manual mapping effort by ~60%.

🧠 Mapping Engine & AI Mapper

Keep the mapper, build the registry, design for template reuse
M1. Keep AI Field Mapper Architecture
Keep

The Error-Code-Only LLM pattern (send only error codes to LLM, never factory data) is correct. Keep the mapping engine and AI mapper as designed.

What Exists:
packages/shared/src/mapping/ — mapping-engine.ts with rule-based + AI-assisted field mapping. Transform rules for unit conversion, date formatting, etc.
M2. Add Template Reuse System to Mapping Engine
Redesign

The mapping engine works for one-off mappings. Redesign it to default to template lookup first, then fall back to AI mapping for new buyer-ERP combinations.

New Flow:
1. Factory connects → identify buyer
2. Check mapping_templates for existing buyer template
3. If golden template exists → apply with factory-specific field name overrides only
4. If no template → AI Mapper generates initial mapping → human reviews → saves as new template
5. Track success rate per template → promote to "golden" after 10 successful uses

This is what makes the second customer faster than the first.
M3. Field Fidelity Test Suite
New

GPT's key insight: "The fatal module to cut is not buyer portal; it is real Tally validation. If you do not prove extraction and field fidelity on live data, everything else is theater."

New Implementation:
A test harness that takes real Tally XML exports and validates: all expected fields present, data types correct, amounts sum correctly, dates parse properly, party names match master data, HSN codes are valid. Run this on every Bridge data push. Report field-level confidence scores. This catches the "weird non-standard Tally customizations" (ledger names like Sales_Local_Export) that Gemini flagged.

🚀 Deployment & Onboarding Design

The "reusable onboarding kit" that GPT called the biggest force multiplier
O1. Factory Onboarding Kit
New

Not a feature — a deployment artifact. This is what makes the second installation faster than the first.

Kit Contents:
1. Pre-flight script — Run on factory PC. Checks: Windows version, port 9000 status, firewall rules, antivirus exclusions needed, Tally version, TDL compatibility, internet speed to FC cloud.
2. Bridge Agent installer — One-click Windows installer (MSI or NSIS). Auto-configures SQLite, sets up service, creates desktop shortcut for health check.
3. TDL package — Pre-built TDL for push events. Installation guide in Telugu and English.
4. Sample mapping pack — Per buyer type. Pre-filled mapping template that factory's accountant can verify field-by-field.
5. Validation test set — 10 sample vouchers per document type. Run through full pipeline. Compare output to expected. Pass/fail report.
6. Rollback procedure — How to uninstall Bridge Agent, restore Tally to previous state, disable TDL.
GPT: "The biggest force multiplier to add is a reusable onboarding kit for one ERP + one buyer + one transaction family." Gemini: "Hire someone who can toggle Port 9000 and talk to the Tally accountant in Telugu."
