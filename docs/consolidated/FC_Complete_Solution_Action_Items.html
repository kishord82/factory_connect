Layer 1 — Integration & Connectivity
MODULE 01 — MVP
ERP Connector Library
RedesignPartial
What Exists
Bridge Agent with Tally XML connector (tally-connector.ts), adaptive poller (adaptive-poller.ts), SQLite local queue (local-queue.ts), cloud sync via WebSocket (cloud-sync.ts). Zoho Books and SAP B1 connector stubs. ERP orchestrator for multi-ERP routing.
Action Items
1. Strip Bridge Agent to read-push-buffer only. All transformation, validation, mapping, and EDI generation moves to cloud. Bridge does: read Tally XML port 9000 → buffer in SQLite → push raw data to cloud. Nothing else.

2. Add TDL push events. Build a custom Tally Definition Language script that fires HTTP POST to Bridge's local endpoint on voucher save/alter/delete. Hybrid approach: TDL push as primary, polling as fallback when TDL isn't installed. This fixes the data latency gap Gemini flagged.

3. Add factory environment health probes. Port 9000 reachability, Tally version detection, TDL install status, Windows firewall check, antivirus interference detection, internet connectivity to FC cloud, SQLite disk space, last successful sync timestamp.

4. Feature-flag Zoho and SAP B1 connectors OFF. Keep the code, disable routing. Tally-only for now. Re-enable when a paying customer needs them.

5. Build Windows installer (MSI/NSIS). One-click install with auto-configure SQLite, auto-create Windows service, desktop shortcut for health check, auto-update capability.
3-Eye Improvement
GPT: "The biggest architectural risk is edge reliability and install friction across messy factory environments."
Gemini: "Port 9000 is notoriously unstable and often blocked by local IT." "Keep the Agent as dumb as possible."
Claude: Design for ugly environments — antivirus quarantine, non-standard TDLs, intermittent connectivity.

Layer 2 — Order & Operations
MODULE 05 — MVP
Order-to-Cash Automation
RedesignPartial
What Exists
Order service (order-service.ts), shipment service, invoice service, resync service. 15-state saga coordinator (saga-service.ts). Outbox poller (outbox-poller.ts). Transactional outbox pattern with 4-write transactions. Canonical models for orders, shipments, invoices.
Action Items
1. CRITICAL: Simplify saga from 15 states to 7. New states: RECEIVED → MAPPED → VALIDATED → APPROVED → SENT → ACKNOWLEDGED → COMPLETED. Plus FAILED, CANCELLED as terminals. Remove: COMPENSATING, COMPENSATED, DEAD_LETTER, RETRYING as separate states. BullMQ handles retries natively. Failed after max retries → exception queue for human review.

2. Keep transactional outbox pattern. Domain write + outbox insert + saga update + audit log in single transaction. All 3 AIs validated this. No change.

3. Keep idempotency keys. Critical for EDI where duplicates cause chargebacks. No change.

4. Add human approval step. Before SENT state, factory export manager reviews and approves the generated document. Simple approve/reject with optional notes. This builds trust during early adoption when factories don't yet trust automated dispatch.

5. Build exception queue. New exception_queue table: document_id, factory_id, error_code, error_detail, assigned_to, status (OPEN/IN_REVIEW/RESOLVED/ESCALATED), resolution_notes. BullMQ worker moves FAILED documents here. Portal shows as a simple list with filters.
3-Eye Improvement
All 3 AIs: Saga is overkill. Simplify. "A retry queue (BullMQ) is enough." — Gemini. "The current vision bundles too many adjacencies too early" — GPT. Keep the good patterns (outbox, idempotency, audit), strip the distributed coordination overhead.

MODULE 18 — MOAT
AI Order Anomaly Detection
Improve DesignNot Started
Action Items
1. Start with simple rule-based anomaly detection, not ML. Flags: quantity > 5x average for this buyer, price > 20% below historical average, new buyer with no history placing large order, delivery date < typical lead time. Rules in app_config table, tunable per factory.

2. Score every incoming order. Risk score 0-100 based on rule matches. Scores above threshold → human review in exception queue. Below threshold → auto-approve. Threshold configurable per factory.

3. Becomes ML-powered with enough data. After 1000+ orders per factory, train simple statistical models (standard deviation from mean). Until then, rules are sufficient and explainable. Don't build ML infrastructure until data volume justifies it.

CROSS-CUT 01
Shadow Mode (GTM Engine)
New Build

All 3 AIs validated Shadow Mode as the strongest GTM asset. It runs the full pipeline (extract → map → validate → generate EDI) WITHOUT dispatching. Compares against what the factory actually sent manually and reports discrepancies.

Action Items
1. Shadow Mode API. POST /api/shadow/enable/:factoryId, GET /api/shadow/report/:factoryId, GET /api/shadow/errors/:factoryId.

2. Dual-processing pipeline. When Shadow Mode is ON, every document goes through: Tally extract → map → validate → generate EDI → compare against manual output → log discrepancies. Never dispatch.
