1. Mission & Product Definition

FactoryConnect is a multi-tenant SaaS platform that connects Indian SME factory ERPs (Tally, Zoho, SAP B1, ERPNext, Busy, Marg) to global buyer procurement systems (EDI X12, SAP Ariba, Coupa, Oracle Fusion, EDIFACT) through a canonical data model and AI-powered mapping engine.

The Problem: Indian factories export $200B+ annually but most run local ERPs (Tally, Busy) with no way to electronically communicate with global buyers who demand EDI/cXML/API integration. Today, this is done manually — emails, phone calls, Excel sheets — leading to errors, delays, and lost business.

The Solution: FC sits between factory and buyer, translating between any source format and any target format. AI auto-maps fields. Configuration replaces coding. One factory can connect to unlimited buyers through one FC subscription.

7. Routing Engine & N×M Resolution

The Routing Engine is a stateless Go service. It reads the connections, routing_rules, and connection_adapter_bindings tables to determine: for this message from this factory, which buyer(s) should receive it, and which target adapter version should process it.

// Core routing: 5 lines of logic, everything else is lookup
connections := getActiveConnections(msg.FactoryID)
for _, conn := range connections {
    binding := getAdapterBinding(conn.ConnectionID)  // Which target adapter?
    mapping := loadMapping(conn.ConnectionID)          // Three-layer mapping
    enqueue("outbound."+binding.TargetAdapterID, TransformJob{msg, mapping, idempotencyKey})
}

Queue-based decoupling: Source adapters → Routing Engine → Redis queue → Target adapters. No direct calls. Natural backpressure. Target adapter restarts don't affect sources.

8. AI Mapping Engine

Claude API analyzes sample data from factories and auto-generates field mappings to the Canonical Model. Three AI tasks:

  1. Source Detection: Given sample data (Tally XML, Zoho JSON, CSV), detect which ERP, which version, and map fields to canonical entities.
  2. Target Detection: Given buyer info, detect which procurement system (EDI X12 version, Ariba cXML version, Coupa API version) and map canonical entities to target format.
  3. Field Matching: For each canonical field, find the corresponding source field and target field. Confidence score per mapping.

Confidence threshold: 90%. Above 90% → auto-applied mapping. Below 90% → flagged for human review. Zero mappings reach production without human verification during onboarding.

10. Three-Layer Mapping Architecture

Layer 1: Base Spec → Layer 2: Buyer Template → Layer 3: Connection Overrides

// Example: Canonical Invoice → EDI X12 810 for Walmart

LAYER 1 — BASE SPEC (X12 5010 standard):
  canonical.invoice_id  →  BIG02 (Invoice Number)
  canonical.order_ref   →  BIG04 (Purchase Order Number)
  canonical.total       →  TDS01 (Total Invoice Amount, in cents)
  canonical.lines[].qty →  IT102 (Quantity Invoiced)

LAYER 2 — BUYER TEMPLATE (Walmart-specific):
  ISA06 qualifier       =  "ZZ"
  ISA08 qualifier       =  "01" (DUNS)
  GS02 sender code      =  factory's assigned Walmart vendor code
  IT106 (Product ID)    =  Walmart GTIN-14 (not factory SKU)
  REF*DP (Department)   =  Walmart department number (REQUIRED)

LAYER 3 — CONNECTION OVERRIDES (Factory A → Walmart):
  ISA06 sender ID       =  "FACTORYCONNECT01"
  GS02 sender code      =  "FC-VENDOR-4521"
  REF*DP value          =  "072" (Factory A ships to Dept 72)
  Custom field mapping: factory's internal SKU → Walmart GTIN lookup table

Change Walmart's template once → propagates to all Walmart connections. Change one factory's specifics at Layer 3 → doesn't affect other factories.

11. Versioned Adapter Lifecycle

Every adapter follows: DEV → ACTIVE → DEPRECATED → SUNSET. The "Two in Production" pattern: deploy new version alongside old. Both run. No forced migrations. Old version decommissioned when zero connections remain for 30 days. Detailed in FC_Versioned_Adapter_Architecture.html.

12. Load-Based Auto-Scaling

Kubernetes HPA for high-traffic adapters (min 1+ pods). KEDA ScaledObjects for low-traffic adapters (scale to zero). Scale metric: Redis queue depth per adapter. Scale-to-zero saves ~30% infra cost. Detailed in FC_Versioned_Adapter_Architecture.html.

15. Three-Way Matching Engine

FC runs three-way matching (PO + Receipt + Invoice) before delivering invoices to buyers. Industry standard tolerances: Quantity ≤3%, Price ≤3% or $100. Exceptions flagged for manual review. This catches errors before the buyer sees them — a major differentiator vs competitors who leave matching to the ERP. Detailed in FC_Integration_Specs_Reference.html.

1. ORIGINAL 10-DECISION FRAMEWORK (from v4)

Each architecture decision was evaluated with Alternatives Rejected and Tradeoffs Accepted before finalizing.

D1: EDI Engine — Data-Driven Spec Engine

  • Alternatives Rejected: Stedi (vendor lock-in, per-transaction cost), custom parser per buyer (N×M problem), EDIFACT-first (Indian buyers don't use it yet)
  • Tradeoffs Accepted: JSON spec maps require manual creation per buyer (~2 days), but eliminate code changes. Spec validation is runtime-only, no compile-time schema checking.
  • Decision Rationale: New buyer = 2 days config, not weeks of code. Spec engine is buyer-agnostic — same engine handles X12, cXML, and future EDIFACT.

D2: AS2 Transport — OpenAS2 Sidecar

  • Alternatives Rejected: Native Node.js AS2 (no mature library), Mendelson AS2 (GPL license), commercial AS2 gateway (cost)
  • Tradeoffs Accepted: Java sidecar adds ~200MB memory, requires JVM. But OpenAS2 is battle-tested for S/MIME signing/encryption/MDN — critical for Walmart compliance.
  • Decision Rationale: AS2 is a solved problem. Don't re-implement cryptographic signing in Node.js when OpenAS2 handles it correctly.

D3: Workflow — BullMQ + Saga + Outbox (No Temporal)

  • Alternatives Rejected: Temporal (complex ops, requires dedicated cluster), AWS Step Functions (vendor lock-in), custom event sourcing (complexity vs team size)
  • Tradeoffs Accepted: Saga coordinator is custom code (~500 lines). No visual workflow editor. But BullMQ is simple, Redis-backed, and the team knows it.
  • Decision Rationale: FC's workflow is a fixed pipeline (PO→ACK→ASN→Invoice), not dynamic branching. BullMQ + outbox + saga is sufficient and operationally simple.

D4: Multi-Tenancy — Row-Level Security

  • Alternatives Rejected: Schema-per-tenant (migration nightmare at 200+ factories), application-level filtering (bug = data leak), separate databases (cost prohibitive)
  • Tradeoffs Accepted: RLS adds ~5% query overhead. Requires discipline: every query must SET LOCAL tenant. But both Gemini AND ChatGPT independently recommended this approach.
  • Decision Rationale: Single schema, single migration, mathematically guaranteed isolation. Security is in the database, not the application.

D5: Buyer Specs — VAN Partnership + Direct Engagement

  • Alternatives Rejected: Scraping public spec docs (incomplete), buying from third parties (expensive, outdated)
  • Tradeoffs Accepted: Requires direct relationship with each buyer's EDI team or their VAN. Slower to onboard first buyer, but specs are authoritative.

D6: AI Mapping — 3-Layer LLM Fallback

  • Alternatives Rejected: Single LLM (single point of failure), manual-only mapping (slow, not scalable), rule-based auto-mapping (brittle)
  • Tradeoffs Accepted: Claude API cost (~₹1,500/month), latency for complex mappings. But 87%+ cache hit rate minimizes actual API calls.
  • Decision Rationale: AI suggests, human confirms. 3 layers ensure mapping never stops: Claude → Gemini/GPT → local model.

D7: EDIFACT — Deferred to Phase 2

  • Tradeoffs Accepted: European buyers can't be served until Phase 2. But spec engine architecture already supports EDIFACT maps — zero architecture change when ready.

D8: Data Deletion — 90-Day Soft-Delete + Legal Hold

  • Alternatives Rejected: Immediate hard delete (compliance risk), unlimited retention (storage cost), factory-uncontrolled retention (regulatory exposure)
  • Tradeoffs Accepted: Dual-threshold (days + size) adds complexity. But gives factories control over their data while meeting DPDP Act requirements.

D9: Infrastructure — OCI Free Tier → Graduated

  • Alternatives Rejected: AWS from day 1 (₹15K+/month minimum), Hetzner (no India region), self-hosted (ops burden)
  • Tradeoffs Accepted: OCI free tier has capacity limits. But 4 OCPU + 24GB RAM handles 50 factories easily. Graduation thresholds are pre-defined.

3. ORDER LIFECYCLE — 15-State Saga

PO_RECEIVED → PO_CONFIRMED → ACK_QUEUED → ACK_SENT → ACK_DELIVERED
  → SHIP_READY → ASN_QUEUED → ASN_SENT → ASN_DELIVERED
  → INVOICE_READY → INVOICE_QUEUED → INVOICE_SENT → INVOICE_DELIVERED
  → COMPLETED

Exception states:
  → FAILED (any step, with retry)
  → COMPENSATION_NEEDED (PO change while ASN in-flight)

Saga table: order_sagas | Column | Type | Purpose | |--------|------|---------| | order_id | FK → canonical_orders | Links to order | | current_step | ENUM(15 states) | Current lifecycle position | | step_deadline | TIMESTAMP | SLA breach time (from connection config) | | locked_by | VARCHAR | Worker ID claiming this step | | lock_expires | TIMESTAMP | Heartbeat expiry (+5 min) | | retry_count | INT | Retries on current step | | compensation_needed | BOOLEAN | PO change detected | | compensation_reason | VARCHAR | Why compensation needed |

Saga poller runs every 60 seconds: 1. Check step_deadline < NOW() → SLA breach alert 2. Check lock_expires < NOW() AND locked_by IS NOT NULL → stale heartbeat, re-enqueue 3. Check compensation_needed = true → notify factory

4. TRANSACTIONAL OUTBOX PATTERN

Every business operation writes domain data + outbox event in a single DB transaction.

confirmOrder(orderId):
  BEGIN;
    UPDATE canonical_orders SET status = 'CONFIRMED';
    INSERT INTO outbox (event_type: 'ORDER_CONFIRMED', aggregate_id: orderId);
    UPDATE order_sagas SET current_step = 'ACK_QUEUED', step_deadline = NOW() + SLA;
    INSERT INTO audit_log (action: 'CONFIRM', ...);
  COMMIT;

Outbox poller (every 5 seconds): 1. SELECT * FROM outbox WHERE processed_at IS NULL ORDER BY created_at LIMIT 100 2. Dispatch to BullMQ with job ID = outbox UUID (idempotent) 3. UPDATE outbox SET processed_at = NOW()

6. AS2 TRANSPORT — OpenAS2 Sidecar

[BullMQ Worker] → writes .edi file → [/data/as2/outbox/{connection}/{msg}.edi]
    ↓
[OpenAS2 Docker container] → picks up → signs (SHA-256) → encrypts (AES-256) → HTTPS POST
    ↓
[Buyer AS2 endpoint] → returns MDN
    ↓
[OpenAS2] → writes MDN → FC parses → updates message_log + saga

Circuit breaker (per connection, Opossum): - Trip at 60% failure rate (minimum 3 requests in 1-minute window) - Half-open after 5 minutes — send test message - Test succeeds → close circuit → drain queued messages - Messages stay in queue during CIRCUIT_OPEN — zero loss

1. Design Philosophy: Think Big, Keep It Simple

Kishor's principle: "Think big and solve with simple — not too complex to build." Every decision in this architecture follows three rules:

1
One Adapter = One Container
Each source/target version is a single Docker container with one job. No shared state, no tangled dependencies.
2
Scale by Load, Not by Hope
Kubernetes HPA scales each adapter independently based on actual message volume. Busy adapters get more pods; idle ones scale to zero.
3
Deploy & Decommission Freely
Add a new version without touching existing ones. Sunset an old version when zero connections remain. No "big bang" migrations.
The Core Insight: FactoryConnect connects N factories (each with different ERP types and versions) to M buyers (each with different procurement systems and versions). Instead of building a monolithic translator that handles every combination, we build small, focused adapters — one per source version, one per target version — and let the Canonical Model in the middle be the universal translator. The routing engine picks the right adapter pair for each message. That's the entire complexity — everything else is just plumbing.

    deprecated := getAdaptersByStatus("DEPRECATED")

7. Routing Engine: Picking the Right Adapter Pair

The Routing Engine is the brain. When a message arrives, it determines which source adapter processed it, which buyer(s) it goes to, and which target adapter version to use for each buyer. It uses the connection_adapter_bindings table for fast lookup.

// Core routing logic — simple and fast
func routeMessage(ctx context.Context, msg CanonicalMessage) error {
    // 1. Find all buyer connections for this factory
    connections := getActiveConnections(msg.FactoryID)

    for _, conn := range connections {
        // 2. Look up bound adapters for this connection
        binding := getAdapterBinding(conn.ConnectionID)

        // 3. Verify target adapter is still ACTIVE or DEPRECATED (not SUNSET)
        adapter := getAdapter(binding.TargetAdapterID)
        if adapter.LifecycleStatus == "SUNSET" {
            // This shouldn't happen, but safety check
            alertOps("Routing to sunset adapter", binding.TargetAdapterID)
            continue
        }

        // 4. Load three-layer mapping for this connection
        mapping := loadMapping(conn.ConnectionID)

        // 5. Enqueue to the target adapter's work queue
        queueName := fmt.Sprintf("outbound.%s", binding.TargetAdapterID)
        enqueue(queueName, TransformJob{
            Message:      msg,
            ConnectionID: conn.ConnectionID,
            Mapping:      mapping,
            IdempotencyKey: msg.IdempotencyKey,
        })
    }
    return nil
}
Why Queue-Based, Not Direct Call: Source adapters don't call target adapters directly. They produce a Canonical Model message, and the Routing Engine enqueues it to the correct target adapter's Redis queue. This decouples source and target completely — a target adapter can be restarted, scaled, or replaced without affecting sources. And it gives us natural backpressure: if a target adapter falls behind, the queue fills, KEDA scales it up.

// internal/security/tls_config.go — shared by all adapters
package security

import (
    "crypto/tls"
    "crypto/x509"
    "os"
)

func NewTLSConfig(requireClientCert bool) *tls.Config {
    cfg := &tls.Config{
        MinVersion: tls.VersionTLS13,   // TLS 1.3 minimum

        // TLS 1.3 cipher preference (Go auto-negotiates, but we set preference)
        CipherSuites: nil, // For TLS 1.3, Go uses the built-in list (cannot be customized)

        // For TLS 1.2 fallback (legacy AS2 buyers only, controlled by feature flag)
        // CipherSuites: []uint16{
        //     tls.TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384,
        //     tls.TLS_ECDHE_RSA_WITH_CHACHA20_POLY1305_SHA256,
        // },

        CurvePreferences: []tls.CurveID{
            tls.X25519,    // Preferred: fastest, most secure
            tls.CurveP256, // Fallback: widely supported
        },

        // Disable session tickets for forward secrecy
        SessionTicketsDisabled: true,
    }

    // mTLS: require client certificate for factory bridge connections
    if requireClientCert {
        caCert, _ := os.ReadFile(secrets.MustGet("FC_CA_CERT_PATH"))
        caCertPool := x509.NewCertPool()
        caCertPool.AppendCertsFromPEM(caCert)
        cfg.ClientCAs = caCertPool
        cfg.ClientAuth = tls.RequireAndVerifyClientCert
    }

    return cfg
}

Recommendation #1: Three-Way Matching (PO + Receipt + Invoice)

This is the #1 automation feature buyers care about. If FC automates three-way matching, it eliminates days of manual reconciliation.

// Three-Way Match Engine (runs on every invoice received)
func threeWayMatch(invoice Invoice) MatchResult {
    // 1. Find the PO
    po := findPO(invoice.POReference)
    if po == nil {
        return MatchResult{Status: "REJECTED", Reason: "PO not found"}
    }

    // 2. Find the receipt (Goods Receipt Note)
    receipt := findReceipt(po.PONumber)
    if receipt == nil {
        return MatchResult{Status: "HOLD", Reason: "No receipt yet — hold for delivery confirmation"}
    }

    // 3. Line-by-line comparison
    for _, invoiceLine := range invoice.Lines {
        poLine := findPOLine(po, invoiceLine.ItemCode)
        receiptLine := findReceiptLine(receipt, invoiceLine.ItemCode)

        // Quantity check: invoice qty ≤ receipt qty ≤ PO qty
        qtyVariance := abs(invoiceLine.Qty - receiptLine.QtyReceived) / receiptLine.QtyReceived
        if qtyVariance > 0.03 {  // 3% tolerance (industry standard)
            return MatchResult{Status: "EXCEPTION", Line: invoiceLine.LineNum,
                Reason: fmt.Sprintf("Qty variance %.1f%% exceeds 3%%", qtyVariance*100)}
        }

        // Price check: invoice price matches PO price
        priceVariance := abs(invoiceLine.UnitPrice - poLine.UnitPrice) / poLine.UnitPrice
        if priceVariance > 0.03 || abs(invoiceLine.UnitPrice - poLine.UnitPrice) > 100 {
            return MatchResult{Status: "EXCEPTION", Line: invoiceLine.LineNum,
                Reason: "Price variance exceeds tolerance (3% or $100)"}
        }
    }

    return MatchResult{Status: "MATCHED", Action: "AUTO_APPROVE_FOR_PAYMENT"}
}

Industry Standard Tolerances: Quantity: 3%. Price: 3% or $100 (whichever is greater). Total: must match within tolerances. Missing line items trigger hold for manual review. This is how Oracle Fusion, SAP Ariba, and Coupa all handle it.

Recommendation #2: Idempotency on Every Mutation

In B2B EDI, messages get retransmitted (network timeouts, AS2 MDN failures, manual retries). Without idempotency, duplicate POs get created. This is non-negotiable.

// Idempotency middleware — applied to every write endpoint
func idempotencyMiddleware(c *gin.Context) {
    key := c.GetHeader("X-Idempotency-Key")
    if key == "" {
        key = c.GetHeader("Idempotency-Key")
    }
    if key == "" {
        c.JSON(400, gin.H{"error": "Idempotency-Key header required for all mutations"})
        c.Abort()
        return
    }

    // Check if we've already processed this key
    cached, err := redis.Get(ctx, "idempotency:"+key).Result()
    if err == nil {
        // Already processed — return the same response
        c.Data(200, "application/json", []byte(cached))
        c.Abort()
        return
    }

    // Acquire distributed lock to prevent concurrent processing
    lock := redis.SetNX(ctx, "idempotency-lock:"+key, "1", 30*time.Second)
    if !lock.Val() {
        c.JSON(409, gin.H{"error": "Request already in progress"})
        c.Abort()
        return
    }

    c.Next()  // Process the request

    // Cache the response for 7 days
    responseBody, _ := c.Get("response_body")
    redis.Set(ctx, "idempotency:"+key, responseBody, 7*24*time.Hour)
    redis.Del(ctx, "idempotency-lock:"+key)
}

Key format: UUID v4. TTL: 7 days. Conflict: Same key + different payload → HTTP 422. This follows Stripe's and Adyen's pattern — the gold standard for payment/order idempotency.

4. AI-Powered Auto-Mapping Flow

This is the intelligence layer that makes onboarding fast. When a factory uploads sample data and a buyer PO, the AI engine performs automatic source/target detection and field mapping.

STEP 1: SOURCE DETECTION Factory uploads sample data (invoice, SO, product list) → AI identifies: format (XML/JSON/CSV/Excel) → AI identifies: ERP type (Tally XML Server response, Zoho webhook JSON, SAP B1 REST) → AI extracts: field names, data types, nesting structure → AI generates: source_mapping (ERP fields → Canonical) → Confidence score: 85-99% for known ERPs, 60-80% for custom STEP 2: TARGET DETECTION Factory uploads sample buyer PO (received from buyer) → AI identifies: buyer system type - Starts with "ISA*" → EDI X12. Extract version from GS08. - Contains "<cXML>" or "<OrderRequest>" → Ariba cXML - JSON with "purchase-order-lines" → Coupa REST - Starts with "UNB+" → EDIFACT. Extract version from UNH. - Other XML/JSON → Custom REST/SOAP. Analyze schema. → AI extracts: buyer identifiers (ISA IDs, ANID, Coupa supplier ID) → AI generates: target_mapping (Canonical → Buyer format) → AI identifies: required acknowledgments (997, ConfirmationRequest) STEP 3: FIELD MATCHING AI matches source fields to target fields through canonical model: Factory "STOCKITEM.NAME" → Canonical "product.name" → EDI "PO1*PID*F*93*" Factory "VOUCHER.AMOUNT" → Canonical "invoice.amount" → EDI "TDS*total*" Factory "GODOWN.CLOSING" → Canonical "inventory.qty" → Ariba "<SupplyQuantity>" For unmatched fields: → AI suggests: closest match with explanation → AI flags: "NEEDS_MANUAL" for ambiguous fields → AI sets: default values for buyer-required fields factory doesn't have STEP 4: VALIDATION AI generates validation rules per connection: → Required fields check (buyer-mandated) → Format validation (date formats, number precision) → Value range checks (quantities, amounts) → Cross-field validation (ship date after order date) → Buyer-specific rules (Walmart requires ASN within 24hrs of shipment)

1. The 14 Grey Areas (Blind Spots)

These are the scenarios that will break in production if not addressed in architecture. Most integration platforms fail here.

Grey Area #1: Source System Field Drift

Problem: Factory updates their ERP (e.g., Tally Prime 4.0 → 4.1). Field names change, new fields appear, old fields deprecated. Zoho renames API fields between versions. SAP B1 patches alter Service Layer responses.

Impact: Existing mappings break silently. Data stops flowing or, worse, maps to wrong fields.

Solution: Source Schema Fingerprinting + Drift Detection

On every data sync, FC computes a schema fingerprint (hash of field names + types + nesting). If the fingerprint changes from the last sync:

  • Auto-pause the connection (don't send corrupted data to buyer)
  • Run AI re-mapping on the new schema against the existing mapping
  • If AI confidence >90%: auto-heal the mapping, log the change, resume
  • If AI confidence <90%: alert FC admin + factory. Manual review required.
  • Store schema history for rollback
// Schema fingerprint on every sync
const currentFingerprint = hashFields(incomingData);
const lastFingerprint = await getLastFingerprint(connectionId);
if (currentFingerprint !== lastFingerprint) {
  await triggerSchemaDriftDetection(connectionId, incomingData, lastFingerprint);
}

Grey Area #2: Buyer EDI Version Mismatch

Problem: Factory's buyer A uses EDI X12 4010, buyer B uses 5010, buyer C uses 5020. Same transaction set (850 PO) but different segment structures, different qualifiers, different mandatory fields. A Walmart 850 looks nothing like a Target 850 even on the same X12 version.

Impact: One-size-fits-all EDI mapping fails. Every buyer needs a buyer-specific implementation guide overlay.

Solution: Three-Layer Mapping Architecture

Layer 1: BASE SPEC (standard X12 4010 / 5010 / Ariba cXML)
  ↓  inherits + overrides
Layer 2: BUYER TEMPLATE (Walmart-specific, H&M-specific, Bosch-specific)
  ↓  inherits + overrides
Layer 3: CONNECTION-LEVEL (factory-specific customizations for this buyer)

// Resolution: connection overrides buyer template overrides base spec
function resolveMapping(connectionId, baseVersion, buyerTemplateId) {
  const base = loadBaseSpec(baseVersion);          // X12 5010 standard
  const buyer = loadBuyerTemplate(buyerTemplateId); // Walmart overlay
  const conn = loadConnectionOverrides(connectionId); // Factory-specific
  return deepMerge(base, buyer, conn); // Connection wins, then buyer, then base
}

This means: when a new factory connects to Walmart, they inherit the Walmart buyer template automatically. Only factory-specific deviations need configuration. The Walmart template is maintained once and shared across all factories selling to Walmart.

Problem: Many factories (especially in Bangladesh, smaller Indian SMEs) don't have any ERP. They run on Excel, paper registers, or WhatsApp. They can't push data to FC via API because there's no system to push from.

Impact: No source connector works. The entire source-side architecture assumes an ERP exists.

Impact: Can't connect to buyers like Bosch (SOAP-based supplier portal), Indian government GSTN (SOAP), older SAP PI/PO systems.

Grey Area #10: Bidirectional Sync Conflicts

Problem: Buyer sends PO. FC creates order in factory ERP. Factory modifies the order in ERP (changes quantity). Buyer also modifies the PO (changes delivery date). Now FC has two conflicting versions. Who wins?

Impact: Data inconsistency. Wrong quantities shipped. Invoice mismatches. Payment disputes.

Solution: Conflict Resolution Policy Engine

Conflict Resolution Rules (configurable per connection):

RULE 1: BUYER_WINS (default for most connections)
  - Buyer is the source of truth for: PO quantity, delivery date, price, ship-to address
  - Factory is the source of truth for: shipment details, invoice data, inventory levels
  - On conflict: buyer's version overwrites, factory notified via WhatsApp

RULE 2: LAST_WRITE_WINS (for equal-authority connections)
  - Timestamp-based resolution. Latest change wins.
  - Both parties notified of the change.

RULE 3: MANUAL_RESOLUTION (for high-value orders)
  - On conflict: pause sync. Alert factory admin. Show diff.
  - Factory must explicitly approve or reject the change.

RULE 4: FIELD-LEVEL RULES (granular)
  - Quantity: buyer_wins
  - Delivery date: buyer_wins
  - Price: factory_wins (factory re-quotes if buyer changes quantity)
  - Packaging instructions: last_write_wins

3. Architecture Recommendations to Make It Bulletproof

#RecommendationWhy & How
1Schema Registry with VersioningEvery source format (Tally XML v3, v4) and target format (X12 4010, 5010) registered as a versioned schema in PostgreSQL. Mappings reference schema versions, not hardcoded formats. When a new version appears, add schema — old mappings continue working.
2Adapter Plugin ArchitectureEach source and target adapter is a self-contained plugin (Node.js module) with a standard interface. New adapters added without touching core engine. Adapters can be hot-loaded without restart. Community/customer can contribute adapters.
3Canonical Model is SacredNEVER add buyer-specific or ERP-specific fields to the canonical model. Use the attributes JSONB field for everything non-standard. The 6 canonical entities are the contract. Changing them breaks everything.
4Message Replay & Audit TrailStore every inbound and outbound message in its raw format (MinIO). Every transformation step logged. If a mapping error is discovered, replay the original messages through the corrected mapping. No data loss ever.
5Circuit Breaker per ConnectionIf a buyer endpoint fails 3 times consecutively, open the circuit breaker. Queue messages. Retry every 5 min. Alert FC ops. Prevents cascade failure where one buyer's outage affects other connections.
6AI Mapping Confidence ThresholdAuto-approve AI mappings with confidence >95%. Send to human review if 80-95%. Reject if <80%. Over time, the threshold increases as AI learns from corrections. Track AI accuracy per ERP type.
7Buyer Template LibraryMaintain pre-built buyer templates for top 50 global buyers (Walmart, Amazon, Bosch, Samsung, H&M, etc.). When a factory says "I sell to Walmart", the entire buyer configuration loads from template. Only factory-specific fields need configuration.
8Backward Compatibility ContractFC API versioning: /v1 supported forever (or until 0 factories use it). /v2 adds features, never removes. Deprecation = 12 months notice + migration assistance. Factory never forced to upgrade.

Summary: How to Build a Better System

The 3 principles that make FactoryConnect resilient across all these grey areas:

1. Configuration over Code: Every new factory-buyer pair is a database row, not a deployment. Generic adapters + AI mapping means zero engineering per connection. Code only changes when adding a fundamentally new protocol.

2. AI as the Onboarding Engine: Claude analyzes sample data, detects systems, generates mappings, suggests defaults, heals schema drift. Humans review, not create. This is what makes 10,000 connections manageable.

3. Version Everything, Break Nothing: Schema registry, versioned adapters, three-layer mapping inheritance, backward-compatible APIs. A factory onboarded today on Tally Prime 3.0 still works when Tally releases version 6.0 — because the old adapter remains.

Coding Standards (Enforced During Development)

  1. DRY Rule — No code block repeated more than 2 times:
  2. If any code block (logic, query pattern, validation, transform, API call, UI component pattern) appears more than twice, it MUST be extracted into a reusable function/utility/hook/component.
  3. Backend examples:
    • DB query wrappers → packages/database/query-helpers.ts (e.g., withTransaction(), findByTenant(), paginatedQuery())
    • Zod parse + error formatting → packages/shared/validation.ts (e.g., parseOrThrow(), formatZodErrors())
    • Outbox event creation → packages/database/outbox-helpers.ts (e.g., enqueueOutboxEvent(tx, type, payload))
    • BullMQ job dispatch pattern → packages/shared/job-helpers.ts (e.g., dispatchJob(queue, type, data, opts))
    • Error response formatting → packages/shared/errors.ts (e.g., throwFcError(code, message, details))
    • Audit log insertion → packages/database/audit-helpers.ts (e.g., auditLog(tx, action, before, after))
    • Redis cache get/set with TTL → packages/shared/cache-helpers.ts (e.g., cacheGetOrFetch(key, ttl, fetcher))
  4. Frontend examples:
    • API fetch + loading + error state → custom hooks (e.g., useApiQuery(), useApiMutation())
    • Form field + label + validation error → reusable <FormField> component
    • Table with filters + pagination + sort → reusable <DataTable> wrapper around TanStack Table
    • Status badge (green/yellow/red) → <StatusBadge status={} /> component
    • Confirmation modal pattern → <ConfirmDialog onConfirm={} message={} />
    • Toast notifications → useToast() hook wrapping a single toast provider
    • Date formatting → utils/date-format.ts (e.g., formatDate(), formatRelative(), toISODate())
  5. Where to put shared code:
    • Cross-package backend utilities → packages/shared/
    • Database-specific helpers → packages/database/
    • Frontend shared components → apps/portal/src/components/common/
    • Frontend hooks → apps/portal/src/hooks/
    • Frontend utilities → apps/portal/src/utils/

  6. ESLint enforcement: Add no-duplicate-code custom rule or use eslint-plugin-sonarjs with no-duplicate-string and no-identical-functions rules to catch violations automatically during make lint.

  7. Naming conventions:

  8. Files: kebab-case.ts (e.g., order-service.ts, sandbox-test-service.ts)
  9. Classes/Types/Interfaces: PascalCase (e.g., CanonicalOrder, MappingConfig)
  10. Functions/variables: camelCase (e.g., createOrder(), tenantId)
  11. Constants: UPPER_SNAKE_CASE (e.g., MAX_RETRY_COUNT, DEFAULT_POLL_INTERVAL_MS)
  12. DB tables/columns: snake_case (e.g., order_lines, factory_id)

  13. API routes: kebab-case (e.g., /api/v1/sandbox-test, /api/v1/mapping-config)

  14. Function size: No function longer than 50 lines. If it is, split into smaller single-responsibility functions.

  15. Error handling pattern: Every thrown error uses FcError class with: error code (FC_ERR_xxx), HTTP status, user-friendly message, internal details. One pattern everywhere — no raw throw new Error().

  16. TypeScript strict mode enforced: strict: true, noImplicitAny: true, noUnusedLocals: true, noUnusedParameters: true. Zero any types — use unknown + type guards instead.

  17. Test co-location: Tests live next to the code they test (e.g., order-service.ts → order-service.test.ts). Not in a separate __tests__/ folder.

  18. Import ordering: External packages → monorepo packages → relative imports. Enforced by eslint-plugin-import.
