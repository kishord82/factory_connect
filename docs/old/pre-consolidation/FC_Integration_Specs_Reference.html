FC Best-Class Recommendation for Tally

Bridge Agent Pattern: Install a lightweight Go binary on the factory's Windows server. It runs as a Windows Service, polls Tally's XML export endpoint every 30s, transforms to FC Canonical Model, and pushes via mTLS to FC cloud. This is how SPS Commerce and TrueCommerce handle Tally-like on-prem ERPs. The bridge agent also receives inbound POs from FC and creates Sales Vouchers in Tally via XML POST. Critical: Always schema-fingerprint every response to detect Tally version drift (field renames between 3.x → 4.x → 7.0).

1.5 Busy 21 ODBC/SQL   +   Marg ERP API Gateway

Busy 21

  • No native API. Zero REST/SOAP endpoints.
  • Database: MS SQL Server (Client-Server mode) or MS Access (File Server mode)
  • Integration: Direct ODBC connection to MSSQL. Read data via SQL queries. Write requires application-layer proxy (RootFi connector).
  • Platform: Windows Only, .NET stack, single-user locking in basic editions
  • Export: Excel, CSV (limited). No webhooks, no real-time sync.
  • FC Strategy: ODBC bridge agent (Windows service) polls SQL tables for new vouchers/invoices. Transforms to canonical model. Push to FC cloud via mTLS. Writing back requires RootFi connector or Busy's file import.

Marg ERP 9.x

  • API Gateway available: Orders, Bills, Stock Status, Outstanding, Collections, Master data.
  • GST/E-Invoice: JWT/JWS standard, SHA256-RSA for digital signatures. Full GST 2.0 compliance.
  • Database: Proprietary (not publicly documented).
  • API Docs: Limited public documentation. Contact: apiintegration@margerp.com
  • Platform: Web + Desktop. More modern than Busy but less documented than ERPNext.
  • FC Strategy: Direct API Gateway integration. Request detailed endpoint documentation from Marg during onboarding. Fallback to Excel/CSV export if API is insufficient.
Key Insight for No-API ERPs (Busy, legacy Tally): FactoryConnect's bridge agent pattern is non-negotiable here. This is how IBM Sterling, SPS Commerce, and TrueCommerce handle on-prem ERPs with no cloud API. The bridge agent sits on the factory's Windows server, reads from the local database/file system, and pushes to the cloud connector. FactoryConnect must ship a lightweight Go binary (~5MB) that runs as a Windows Service — zero dependencies, auto-updates, heartbeat monitoring.

2.1 EDI X12 EDI/AS2
AttributeDetails
Versions in Use4010 (legacy, still common), 5010 (current standard, HIPAA-mandated for healthcare), 5020, 5030, 8020. 5010 is the target baseline.
Envelope StructureISA/IEA (Interchange, fixed 106 chars) → GS/GE (Functional Group) → ST/SE (Transaction Set). Delimiters: * (element), ~ (segment), : (sub-element).
Core Transaction Sets850 (PO), 855 (PO Ack), 856 (ASN/Ship Notice), 810 (Invoice), 820 (Payment/Remittance), 860 (PO Change), 997 (Functional Ack), 999 (Implementation Ack).
ISA Qualifiers01=DUNS, 12=Phone, 14=DUNS+4, 20=EIN, ZZ=Mutually Defined. GS codes: PO=Purchase Order, IN=Invoice, SH=Ship Notice.
Validation Rules5 syntax rule types: R (Required), E (Exclusive), L (List Conditional), C (Conditional), P (Paired). Mandatory + conditional segments per implementation guide.
997 AcknowledgmentConfirms receipt + syntactical correctness. AK5 status: A=Accepted, E=Accepted w/Errors, R=Rejected. One 997 per received functional group.
Buyer-SpecificWalmart: AS2 mandatory, zero-tolerance errors, rigid shipping labels. Target: strict delivery timelines. Costco: three-way match (PO+Shipment+Invoice must align perfectly). Each buyer has custom qualifiers, loops, and code tables.

2.2 AS2 Protocol (RFC 4130) AS2/S/MIME
AttributeDetails
How It WorksHTTP POST with S/MIME encryption + digital signature. Peer-to-peer, no intermediaries. Can carry X12, EDIFACT, XML, or any structured data.
Key HeadersAS2-From, AS2-To, Message-ID (globally unique), Content-Type: application/edi-x12
EncryptionAES-256 (recommended), AES-128, 3DES (legacy — FC will not support 3DES). S/MIME encrypts payload; TLS encrypts transport.
SigningSHA-256 (recommended minimum), SHA-384, SHA-512. SHA-1 deprecated but supported for legacy partners. RSA-2048+ keys.
MDNSynchronous (same HTTP response, fast) or Asynchronous (separate HTTP/SMTP, for large files). Signed MDN provides non-repudiation. AWS Transfer Family supports async MDN as of March 2026.
CertificatesRSA 2048-bit minimum (CA-signed recommended for production). Key usage: Digital Signature + Data Encipherment. Some buyers mandate CA-signed.
CompressionZLIB/DEFLATE (RFC 5402). 60-80% compression for EDI documents. Optional, per-partner.
AWS Transfer FamilyManaged AS2 service. Drummond certified. Auto-upload to S3. CloudFormation support. Async MDN (March 2026). Certificate monitoring via DescribeCertificate API.

FC Best-Class Recommendation for AS2

Use AWS Transfer Family for AS2 transport (as identified in the Cloud Hosting Comparison). This gives us managed, Drummond-certified AS2 without building our own AS2 stack. FC handles the message creation (EDI segments) and AWS handles the transport (encryption, signing, MDN, delivery). Estimated cost: $0.10/message + transfer fees. For 500 factories × 10 msgs/day = ~$15K/year. Much cheaper than building and maintaining our own AS2 infrastructure.

Part 4: Industry Best-Class Recommendations

Recommendation #1: Three-Way Matching (PO + Receipt + Invoice)

This is the #1 automation feature buyers care about. If FC automates three-way matching, it eliminates days of manual reconciliation.

// Three-Way Match Engine (runs on every invoice received)
func threeWayMatch(invoice Invoice) MatchResult {
    // 1. Find the PO
    po := findPO(invoice.POReference)
    if po == nil {
        return MatchResult{Status: "REJECTED", Reason: "PO not found"}
    }

    // 2. Find the receipt (Goods Receipt Note)
    receipt := findReceipt(po.PONumber)
    if receipt == nil {
        return MatchResult{Status: "HOLD", Reason: "No receipt yet — hold for delivery confirmation"}
    }

    // 3. Line-by-line comparison
    for _, invoiceLine := range invoice.Lines {
        poLine := findPOLine(po, invoiceLine.ItemCode)
        receiptLine := findReceiptLine(receipt, invoiceLine.ItemCode)

        // Quantity check: invoice qty ≤ receipt qty ≤ PO qty
        qtyVariance := abs(invoiceLine.Qty - receiptLine.QtyReceived) / receiptLine.QtyReceived
        if qtyVariance > 0.03 {  // 3% tolerance (industry standard)
            return MatchResult{Status: "EXCEPTION", Line: invoiceLine.LineNum,
                Reason: fmt.Sprintf("Qty variance %.1f%% exceeds 3%%", qtyVariance*100)}
        }

        // Price check: invoice price matches PO price
        priceVariance := abs(invoiceLine.UnitPrice - poLine.UnitPrice) / poLine.UnitPrice
        if priceVariance > 0.03 || abs(invoiceLine.UnitPrice - poLine.UnitPrice) > 100 {
            return MatchResult{Status: "EXCEPTION", Line: invoiceLine.LineNum,
                Reason: "Price variance exceeds tolerance (3% or $100)"}
        }
    }

    return MatchResult{Status: "MATCHED", Action: "AUTO_APPROVE_FOR_PAYMENT"}
}

Industry Standard Tolerances: Quantity: 3%. Price: 3% or $100 (whichever is greater). Total: must match within tolerances. Missing line items trigger hold for manual review. This is how Oracle Fusion, SAP Ariba, and Coupa all handle it.

Recommendation #2: Idempotency on Every Mutation

In B2B EDI, messages get retransmitted (network timeouts, AS2 MDN failures, manual retries). Without idempotency, duplicate POs get created. This is non-negotiable.

// Idempotency middleware — applied to every write endpoint
func idempotencyMiddleware(c *gin.Context) {
    key := c.GetHeader("X-Idempotency-Key")
    if key == "" {
        key = c.GetHeader("Idempotency-Key")
    }
    if key == "" {
        c.JSON(400, gin.H{"error": "Idempotency-Key header required for all mutations"})
        c.Abort()
        return
    }

    // Check if we've already processed this key
    cached, err := redis.Get(ctx, "idempotency:"+key).Result()
    if err == nil {
        // Already processed — return the same response
        c.Data(200, "application/json", []byte(cached))
        c.Abort()
        return
    }

    // Acquire distributed lock to prevent concurrent processing
    lock := redis.SetNX(ctx, "idempotency-lock:"+key, "1", 30*time.Second)
    if !lock.Val() {
        c.JSON(409, gin.H{"error": "Request already in progress"})
        c.Abort()
        return
    }

    c.Next()  // Process the request

    // Cache the response for 7 days
    responseBody, _ := c.Get("response_body")
    redis.Set(ctx, "idempotency:"+key, responseBody, 7*24*time.Hour)
    redis.Del(ctx, "idempotency-lock:"+key)
}

Key format: UUID v4. TTL: 7 days. Conflict: Same key + different payload → HTTP 422. This follows Stripe's and Adyen's pattern — the gold standard for payment/order idempotency.

Recommendation #3: Cascade Document Dependencies

The #1 error in EDI implementations is sending documents out of order. An ASN (856) without a confirmed PO Ack (855) gets rejected. An invoice (810) without a receipt gets held.

// Document dependency chain — enforced by routing engine
DOCUMENT DEPENDENCIES (enforced, not advisory):

  EDI 850 (PO) received
    ├── MUST send 997 (Functional Ack) within 1 hour
    ├── MUST send 855 (PO Ack) within 24-48 hours
    │   ├── BLOCKED: Cannot send 856 until 855 is confirmed by buyer
    │   └── If buyer rejects 855 → resolve before proceeding
    ├── MUST send 856 (ASN) before/on day of delivery
    │   ├── BLOCKED: Cannot send 810 until 856 is delivered
    │   └── Must include SSCC-18 labels if buyer requires
    └── MUST send 810 (Invoice) after goods delivery
        ├── Three-way match must pass (PO + Receipt + Invoice)
        └── Price must match PO ± tolerance

// State machine per order (no skipping allowed)
PO_RECEIVED → PO_ACKNOWLEDGED → SHIPPED → INVOICED → PAID → COMPLETED
                                                     ↘ DISPUTED (if match fails)

Recommendation #4: Pre-Validation Gate (Before Touching ERP)

Never push an invalid PO into the factory's ERP. Validate everything in FC first.

// 6-step validation before creating Sales Order in factory ERP

1. STRUCTURAL VALIDATION
   ├─ PO number unique (not a duplicate)
   ├─ PO date valid (not in past, not future beyond 1 year)
   ├─ All mandatory fields present (buyer ID, items, quantities, prices)
   └─ XML/JSON/EDI schema compliance

2. MASTER DATA VALIDATION
   ├─ Buyer exists in FC system and is active
   ├─ Item codes exist in factory's catalog
   ├─ Delivery address is valid
   └─ Currency is supported

3. BUSINESS LOGIC VALIDATION
   ├─ Quantities within min/max order limits
   ├─ Prices match agreed contract prices (from mapping config)
   ├─ Total PO value within buyer's credit limit
   ├─ Payment terms are acceptable
   └─ Delivery date is achievable (lead time check)

4. INVENTORY VALIDATION (if factory ERP supports real-time stock)
   ├─ Stock available or backorder acceptable
   ├─ Fulfillment ratio (can meet 50%+ of order?)
   └─ Warehouse assignment

5. COMPLIANCE VALIDATION
   ├─ HSN/SAC codes valid (India GST)
   ├─ CBAM data present if EU buyer
   └─ Export documentation requirements met

6. RESULT → Accept / Accept with Changes / Reject
   ├─ ACCEPT: Create Sales Order in factory ERP
   ├─ ACCEPT WITH CHANGES: Create with modifications, notify buyer
   └─ REJECT: Send rejection reason to buyer

  1. AI-Powered Auto-Mapping — No other platform auto-detects source fields, target formats, and generates mappings using AI. This is FC's moat.
