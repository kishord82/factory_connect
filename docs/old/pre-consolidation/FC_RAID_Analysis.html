1.1 Technical Risks
ID Severity Risk P × I Description & Impact Mitigation Strategy
R-T01 CRITICAL ERP API Breaking Changes 4 × 5 = 20 Source ERPs (Tally, Zoho, SAP B1, ERPNext) release new versions that break existing adapter integrations. TallyPrime 7.0 added JSON alongside XML — future versions may deprecate XML. Zoho Books moved from v2→v3 with breaking changes. SAP B1 is migrating OData 3→4 at FP2405. ERPNext v15 broke custom field naming conventions. Impact: Data flow stops for affected factories until adapter is updated. SLA breach, customer churn. 1. Version Monitoring Service: Automated crawler checks ERP vendor changelogs/release notes weekly. Alerts on any mention of deprecation, breaking change, or new API version.
2. Two-in-Production Pattern: Always run current and next version adapters simultaneously. New factories onboard to latest; existing connections migrate on schedule.
3. Adapter Abstraction Layer: Each adapter implements a canonical interface. Internal logic changes don't ripple outward.
4. 90-Day Migration Window: From detection of breaking change to full cutover, target 90 days.
Owner: Engineering Lead
R-T02 CRITICAL Buyer EDI Rejection Cascade 4 × 5 = 20 Major buyers (Walmart, Target, Costco) have zero-tolerance EDI validation. A single malformed segment (wrong qualifier, missing element, bad date format) triggers 997 rejection. If FC produces even 2-3% rejection rate, buyers will blocklist the factory's ANID/ISA qualifier. Impact: Factory loses buyer relationship. FC loses customer and reputation. Potential legal liability. 1. Pre-Flight Validation Engine: Every outbound EDI document passes through a multi-layer validator: structural (segment counts, delimiters), semantic (code values, date ranges), buyer-specific (Walmart Implementation Guide rules).
2. Buyer-Specific Test Harness: Mock buyer endpoints that replicate exact validation rules. 100% of test cases must pass before production go-live.
3. Shadow Mode: First 2 weeks of any new connection run in shadow mode — FC generates documents but doesn't send. Human review before switching to live.
4. Rollback Circuit Breaker: If rejection rate exceeds 1% in any 1-hour window, circuit breaker halts outbound and alerts ops team.
Owner: Integration Lead
R-T03 CRITICAL Canonical Model Inadequacy 3 × 5 = 15 The 6-entity canonical model (Product, Inventory, Order, Shipment, Invoice, Party) may not capture all required fields for every source/target combination. Industry-specific fields (textile lot numbers, pharma batch codes, automotive VIN-linked parts) may not fit the model. Impact: Data loss during translation. Factory can't fulfill buyer requirements. Must redesign core model — affects every adapter. 1. Extension Fields: Canonical model includes a custom_fields map[string]interface{} for industry-specific data that passes through without transformation.
2. Canonical Model Versioning: Model itself is versioned (v1, v2). Adapters declare which version they support. Backward-compatible additions don't break existing adapters.
3. Pre-Launch Field Audit: Before onboarding any new industry vertical, audit all required fields against canonical model. Add to model before building adapters.
4. Buyer Implementation Guide Review: Parse every buyer's implementation guide to extract required fields. Map each to canonical field or flag as gap.
Owner: Architect
R-T04 HIGH AI Mapping Accuracy Below Threshold 3 × 4 = 12 The AI mapping engine (Claude API) auto-maps source ERP fields to canonical model. If accuracy is below 90%, every onboarding requires extensive manual mapping review, destroying the "configuration over coding" value proposition. LLM hallucination could map wrong fields silently. Impact: Onboarding time increases 5-10x. Cost per customer rises. "Simple configuration" promise broken. 1. Confidence Scoring: Every AI mapping includes a confidence score (0-1). Mappings below 0.85 are flagged for human review. Only auto-approve above 0.95.
2. Mapping Templates: Pre-built mapping templates for common ERP+buyer combinations (Tally→Walmart EDI, Zoho→Ariba cXML). AI refines templates, doesn't start from scratch.
3. Mapping Validation Suite: Test each mapping with real sample data before activation. Compare output to expected output for 10+ test documents.
4. Feedback Loop: Every manual correction trains the model. After 50+ corrections per ERP type, accuracy should exceed 95%.
Owner: AI Engineering Lead
R-T05 HIGH Rate Limiting Causes Data Delays 4 × 3 = 12 Source ERPs have strict rate limits: Zoho 100 req/min (strictest), Tally 100 req/min, SAP B1 session-based with 30min timeout. A factory with 500+ orders/day could saturate the rate limit. Impact: Order processing delays. Buyer SLAs missed. Cascading backpressure through the system. 1. Per-Connection Rate Limiter: Token bucket algorithm per connection, configured to 80% of ERP's stated limit. Never hit the hard limit.
2. Batch Aggregation: Aggregate multiple small requests into batch calls where ERP supports it (Zoho batch API, SAP B1 batch OData).
3. Webhook-First Strategy: For ERPs that support webhooks (Zoho, ERPNext), use push model instead of polling. Eliminates most polling requests.
4. Prioritized Queue: When near rate limit, prioritize time-sensitive documents (PO Acknowledgments, ASNs) over less urgent (reports, inventory syncs).
Owner: Platform Engineering
R-T06 HIGH Message Ordering & Exactly-Once Delivery Failures 3 × 4 = 12 Order-to-Cash flow has strict cascade dependencies: PO → PO Ack → ASN → Invoice → Payment. If messages arrive out of order or are duplicated (network retry), downstream documents reference non-existent predecessors. Impact: Buyer systems reject documents. Three-way match fails. Payment delayed. 1. Idempotency Keys: UUID-based idempotency on every mutation. 7-day TTL in Redis. Duplicate detection at ingress.
2. Cascade Dependency Engine: Each document type declares its prerequisites. Engine holds documents until prerequisites are confirmed delivered and acknowledged.
3. Sequence Numbers: Per-connection monotonic sequence numbers. Receiver detects gaps and requests retransmission.
4. Dead Letter Queue + Human Escalation: After 3 retries, move to DLQ. Alert ops team within 15 minutes. SLA: DLQ items resolved within 4 hours.
Owner: Platform Engineering
R-T07 HIGH Bridge Agent Deployment & Maintenance Burden 4 × 3 = 12 Tally and Busy are Windows-only, requiring a bridge agent installed on factory's local network. Indian SME factories may have unreliable IT infrastructure: outdated Windows versions, no IT staff, intermittent internet, firewall restrictions. Impact: Onboarding cost is 3-5x higher for Tally/Busy factories. Support burden for on-premise agent issues. 1. Auto-Updating Agent: MSI installer with Windows Service. Auto-updates via signed manifests. Rollback on failure.
2. Heartbeat Monitoring: Agent sends heartbeat every 60 seconds. If missed for 5 minutes, alert FC ops + factory contact via WhatsApp.
3. Offline Buffer: Agent buffers messages locally (SQLite) during internet outages. Syncs when connection restores. Buffer holds 72 hours of data.
4. Remote Diagnostics: Agent exposes diagnostic endpoint (local only). FC support can request diagnostics via secure tunnel (Cloudflare Tunnel or Tailscale).
5. Partner Network: Build relationships with local IT service providers in Indian industrial cities for on-site support.
Owner: DevOps + Customer Success
R-T08 HIGH Multi-Tenant Data Isolation Breach 2 × 5 = 10 As a multi-tenant SaaS, a bug in tenant isolation could expose Factory A's data (orders, invoices, pricing) to Factory B. Shared database with tenant_id filtering is vulnerable to missing WHERE clauses. Impact: Catastrophic — customer trust destroyed. Legal liability. Potential regulatory fines. 1. Row-Level Security (RLS): PostgreSQL RLS policies enforced at database level. Even if application code misses tenant_id, database blocks cross-tenant access.
2. Tenant Context Middleware: Every request sets tenant context at gateway level. Propagated via request headers. All DB queries inject tenant filter from context, not from user input.
3. Automated Isolation Tests: CI pipeline includes tests that attempt cross-tenant access on every endpoint. Build fails if any succeed.
4. Encryption at Rest per Tenant: Each tenant's sensitive data (credentials, mappings) encrypted with tenant-specific key derived from master key.
Owner: Security Lead
R-T09 MEDIUM Kubernetes Complexity Overhead 3 × 3 = 9 Running 25+ adapter microservices on Kubernetes with HPA/KEDA requires significant DevOps expertise. Early-stage startup may not have dedicated SRE. Misconfigured HPA can lead to pod thrashing or under-provisioning. Impact: Operational instability. High infrastructure cost if auto-scaling is misconfigured. 1. Managed Kubernetes: Use GKE Autopilot to offload node management, security patches, and scaling of the cluster itself.
2. GitOps with ArgoCD: All K8s manifests in git. Changes via PR review. ArgoCD syncs automatically. Reduces manual kubectl errors.
3. Progressive Rollout: Start with 5-8 adapters in Phase 1. Add complexity only as customer base grows. Don't deploy all 25 adapters day one.
4. Infrastructure as Code: Terraform for all cloud resources. Pulumi for Kubernetes resources. Reproducible from scratch in <2 hours.
Owner: DevOps Lead
R-T10 MEDIUM Database Schema Migration at Scale 3 × 3 = 9 As the platform evolves, schema migrations must run across a multi-tenant database without downtime. Migrations that lock tables (ALTER TABLE ADD COLUMN with default) can block all tenant operations. Impact: Service outage during migration. Data inconsistency if migration partially fails. 1. Online Schema Migration: Use pg_repack or GitHub's gh-ost pattern for zero-downtime ALTER TABLEs.
2. Expand-Contract Pattern: Add new column → deploy code that writes to both → backfill → deploy code that reads from new → drop old column. Never destructive in one step.
3. Migration Testing: Every migration tested against a production-sized dataset copy before deployment.
4. Rollback Scripts: Every UP migration has a corresponding DOWN. Tested in CI.
Owner: Database Lead
R-T11 MEDIUM Redis as Single Point of Failure for Queues 3 × 3 = 9 All inter-adapter routing goes through Redis queues. If Redis goes down, all message processing halts. Redis persistence (RDB/AOF) can lose recent data on crash. Impact: Complete message processing halt. Potential message loss. 1. Redis Sentinel or Cluster: Minimum 3-node Redis Sentinel for automatic failover. Promote replica within seconds.
2. AOF with fsync=everysec: At most 1 second of data loss on crash. Acceptable for queue messages that can be retried from source.
3. Dual-Write to PostgreSQL: Every message enqueued to Redis is also recorded in PostgreSQL message_log. On Redis recovery, replay unprocessed messages from PostgreSQL.
4. Consider NATS/Kafka Migration (Phase 3): If message volume exceeds 10K/min, evaluate migration to NATS JetStream or Kafka for stronger durability guarantees.
Owner: Platform Engineering
R-T12 MEDIUM Observability Blind Spots in Distributed Tracing 3 × 3 = 9 A single order flows through 4-6 microservices (source adapter → routing → canonical transform → target adapter → acknowledgment). If tracing breaks at any hop, debugging production issues becomes guesswork. Impact: Slow incident resolution. MTTR increases. Customer frustration. 1. OpenTelemetry SDK in Every Service: Mandatory. Trace ID propagated via W3C Trace Context headers. No exceptions.
2. Correlation IDs: Every message carries trace_id, connection_id, document_id. Searchable in Jaeger and logs.
3. Structured Logging: JSON logs with correlation IDs. Shipped to centralized logging (Loki or CloudWatch Logs).
4. SLO Dashboards: Per-connection success rate, latency P95, error rate. Alert on SLO breach, not just errors.
Owner: SRE
R-T13 MEDIUM Go-to-Python Inter-Service Latency 2 × 3 = 6 AI gateway runs Python/FastAPI while all other services run Go. Cross-language gRPC/HTTP calls between Go services and Python AI gateway add serialization overhead and potential version mismatch in protobuf schemas. Impact: Onboarding latency. Additional deployment complexity. 1. Async AI Calls: AI mapping is not in the hot path. Onboarding runs asynchronously. Latency is acceptable for one-time mapping.
2. Cache AI Results: Once a mapping is computed, cache it permanently. AI is called only during onboarding, not per-message.
3. Shared Protobuf Contracts: Single proto file repo. CI generates Go and Python stubs. Version mismatch caught at build time.
Owner: Engineering Lead
R-T14 LOW Go Module Dependency Vulnerabilities 2 × 2 = 4 Go microservices depend on third-party modules (Gin, GORM, go-redis, etc.). Vulnerabilities in dependencies could expose the platform. Impact: Security vulnerability. Emergency patching required. 1. Dependabot/Renovate: Automated PRs for dependency updates. Weekly cadence.
2. govulncheck in CI: Go's official vulnerability checker runs on every build. Build fails on known CVEs.
3. Minimal Dependencies: Prefer stdlib where possible. Audit any new dependency before adding.
Owner: Engineering Lead

1.4 Security & Compliance Risks
ID Severity Risk P × I Description & Impact Mitigation Strategy
R-S01 HIGH AS2 Certificate Management Failures 3 × 4 = 12 AS2 transport (used by Walmart, Target, and most large US buyers) requires S/MIME encryption with RSA-2048+ certificates. Certificates expire. Certificate rotation with 500+ buyer connections is operationally complex. A single expired cert = all documents to that buyer fail. Impact: Silent data delivery failures. Buyer relationship damage. 1. Certificate Inventory Database: Track every certificate: issuer, expiry date, key size, associated connections. Alert 90/60/30 days before expiry.
2. Automated Renewal: For FC-owned certs, use Let's Encrypt or Sectigo with auto-renewal. For buyer-provided certs, automated reminder workflow.
3. Certificate Health Check: Daily job validates every active certificate. Alerts on approaching expiry, weak key sizes, or revocation.
4. Mutual Authentication Testing: Monthly automated test that establishes mTLS/AS2 connection to every buyer endpoint. Catches issues before they affect production.
Owner: Security Lead
R-S02 HIGH Data Residency & GDPR/DPDP Compliance 3 × 4 = 12 Indian factories' data must comply with India's Digital Personal Data Protection Act (DPDP 2023). EU buyers may require GDPR compliance. Data residency requirements may mandate data stays in specific regions. FC processes business data (orders, invoices) that may contain personal data (contact names, addresses). Impact: Regulatory fines. Cannot serve EU buyers. Legal liability. 1. Data Classification: Classify all data fields: public, confidential, personal, sensitive. Apply appropriate controls per classification.
2. Regional Deployment Capability: Architecture supports deploying in GCP Mumbai (asia-south1) for India, GCP Belgium (europe-west1) for EU. Data stays in region.
3. Data Processing Agreement Templates: Standard DPA for factories. GDPR-compliant DPA for EU buyers.
4. Legal Review (Phase 2): Engage data privacy lawyer for DPDP and GDPR compliance audit before EU buyer onboarding.
Owner: CEO + Legal
R-S03 MEDIUM Credential Storage for Factory ERPs 3 × 4 = 12 FC must store factory's ERP credentials (API keys, OAuth tokens, ODBC passwords) to poll data. If FC's credential store is breached, attacker has access to hundreds of factory ERP systems. Impact: Mass breach of customer systems. Existential threat to company. 1. HashiCorp Vault (Production): All credentials in Vault with auto-seal, audit logging, and lease-based access. No credentials in environment variables or config files in production.
2. Least-Privilege Credentials: Factory provides read-only API keys. FC never needs write access to factory ERP (except for status updates, which use separate limited-scope credentials).
3. Credential Rotation Policy: Automated 90-day rotation for OAuth tokens. Alert on credentials older than 90 days.
4. Network Isolation: Vault in private subnet. Access only from service accounts via mTLS. No internet exposure.
Owner: Security Lead
R-S04 MEDIUM SOC 2 Type II Requirement from Enterprise Buyers 3 × 3 = 9 Enterprise buyers (Walmart, Amazon Vendor Central) may require SOC 2 Type II certification before approving FC as a supplier integration platform. SOC 2 audit takes 6-12 months and costs $50-150K. Impact: Blocked from enterprise buyer onboarding. Limits addressable market to mid-market buyers. 1. SOC 2 Readiness (Phase 1): Implement SOC 2-aligned controls from day one (access controls, audit logging, encryption, incident response). Don't defer security to later.
2. SOC 2 Type I First: Type I (point-in-time) is faster and cheaper. Achievable in 3-4 months. Satisfies many buyers while working toward Type II.
3. Vanta/Drata: Use compliance automation platforms to continuously monitor SOC 2 controls. Reduces audit cost and preparation time by 60%.
Owner: CEO + Security Lead

A-T01 CRITICAL Tally XML API Will Remain Available Tally's XML-over-HTTP interface (port 9000) is FC's primary integration point for India's most popular ERP. TallyPrime 7.0 added JSON but didn't deprecate XML. We assume XML will remain available for 3+ years. If Tally removes XML or moves to a proprietary cloud-only API, FC's largest source adapter becomes useless. Monitor Tally developer portal quarterly. Track TallyPrime release notes. Attend Tally partner conferences. Build relationship with Tally integration team. Build JSON adapter as parallel path. If XML deprecated, migrate factories to JSON within 90 days using Two-in-Production pattern. If Tally goes cloud-only, build cloud API adapter.
A-T02 CRITICAL Canonical Model Covers 90%+ of Use Cases The 6-entity canonical model (Product, Inventory, Order, Shipment, Invoice, Party) covers the standard Order-to-Cash flow. We assume this covers 90%+ of real-world factory-to-buyer data exchange. Untested with industry-specific flows (pharma, automotive, defense). Onboard 5+ factories from 3+ different industries. Track % of fields that fit canonical model vs. need custom_fields extension. Target: >90% canonical coverage. If canonical model covers only 70-80%, add industry-specific entity extensions (PharmaProduct, AutoPart). Each extension is a superset of base entity. Adds complexity but preserves core architecture.
A-T03 CRITICAL EDI X12 4010 is Sufficient for Most US Buyers EDI X12 version 4010 (released 1997) is still the most widely used version. We build 4010 first and assume it covers 80%+ of US buyer requirements. Some buyers may mandate 5010 or 5020. Survey first 20 target buyers for their EDI version requirements. Track buyer implementation guides to determine version distribution. Build 5010 adapter in Phase 2. Architecture already supports versioned adapters. Buyer connections can be routed to version-specific adapters.
A-T04 HIGH Zoho/SAP B1/ERPNext APIs Won't Change Rate Limits Downward Current rate limits: Zoho 100 req/min, SAP B1 session-based, ERPNext no hard limit. We've designed around these limits. If vendors tighten limits (e.g., Zoho drops to 50 req/min), high-volume factories hit bottlenecks. Monitor vendor changelog monthly. Track actual API response headers for rate limit changes (X-RateLimit headers). Alert on any degradation. Implement aggressive caching + webhook-first strategy. Reduce polling frequency. Batch requests. If extreme, negotiate higher limits with vendor (paid API tier).
A-T05 HIGH Claude API Remains Cost-Effective for AI Mapping AI mapping engine uses Claude API for field mapping and auto-configuration. We assume Claude API pricing remains stable (~$3/MTok input, $15/MTok output for Opus) and availability remains high (99.9%+). Track Claude API cost per onboarding. Monitor Anthropic pricing announcements. Benchmark against GPT-4, Gemini quarterly. Build abstraction layer that supports multiple LLM providers. If Claude pricing doubles, switch to most cost-effective provider. AI mapping is called only during onboarding, not per-message — cost impact is bounded.
A-T06 HIGH GKE Autopilot Handles Our Scaling Pattern We assume GKE Autopilot can handle the "many small pods scaling independently" pattern (25+ deployments, KEDA scale-to-zero, HPA). Autopilot manages nodes automatically but has constraints on pod configurations. Load test with k6 simulating 100 concurrent connections across 15 adapter types. Verify HPA/KEDA behavior under Autopilot constraints. Fall back to GKE Standard with managed node pools. More operational overhead but full control over node configuration and scheduling.
A-T07 MEDIUM Redis Can Handle Queue Volume at Scale All inter-adapter message routing goes through Redis Streams/Lists. We assume Redis can handle 10K+ messages/min with acceptable latency (<10ms). Beyond that, we'd need Kafka or NATS. Load test Redis Cluster with simulated production traffic pattern. Measure latency at P50, P95, P99 at various throughput levels. Migrate to NATS JetStream or Kafka for message routing. Redis continues as cache layer. Migration path designed into architecture (queue interface abstraction).
A-T08 MEDIUM Three-Way Matching Tolerances Are Universal We've set default tolerances: 3% quantity, 3%/$100 price variance. Assumption: these work for most buyer-factory pairs. Some buyers may have zero tolerance or different thresholds. Collect actual tolerance requirements from first 20 buyer connections. Compare to our defaults. Make tolerances configurable per-connection in the Three-Layer Mapping (Connection Override level). Already architecturally supported.
A-T09 MEDIUM Go + Gin is the Right Tech Stack We chose Go for all microservices due to performance, goroutine concurrency, and small binary sizes. Assumption: Go ecosystem has mature libraries for EDI parsing, XML processing, cXML generation, and AS2 signing. Build proof-of-concept EDI 850 parser, AS2 signer, and cXML generator in Go.
