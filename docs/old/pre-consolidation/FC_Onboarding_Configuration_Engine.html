FactoryConnect — Onboarding Configuration Engine

End-to-end specification: What the factory provides, what FC provisions, what the buyer provides, and how AI auto-maps the connection for N factories × M buyers | April 2026

1. The Missing Piece: Bidirectional Provisioning

The current architecture defines the routing tables (connections, routing_rules, mapping_configs) but doesn't specify the onboarding workflow — the step-by-step provisioning where factory, buyer, and FactoryConnect each contribute configuration to establish a live connection. This document fills that gap.

3. The Onboarding Configuration Database Schema

This extends the existing connections table with a full onboarding state machine and configuration registry:

-- Onboarding configuration per factory-buyer connection
CREATE TABLE connection_configs (
  config_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  connection_id UUID NOT NULL REFERENCES connections(connection_id),

  -- FACTORY SIDE CONFIG (what factory provided)
  factory_erp_config JSONB NOT NULL,      -- {erp_type, version, api_url, auth_type, polling_interval}
  factory_erp_credentials JSONB,          -- ENCRYPTED: {oauth_token, api_key, username, password}
  factory_buyer_code VARCHAR NOT NULL,    -- Factory's internal code for this buyer in ERP
  factory_sample_data JSONB,             -- References to uploaded sample files (S3/MinIO paths)
  factory_custom_fields JSONB,           -- Industry-specific field definitions
  factory_network_config JSONB,          -- {static_ip, proxy, firewall_type, whitelisted_ips}

  -- FC PROVISIONS (what FC generated/assigned)
  fc_api_key VARCHAR NOT NULL,            -- FC-issued API key for this factory
  fc_api_endpoint VARCHAR NOT NULL,       -- Dedicated or shared API endpoint
  fc_webhook_url VARCHAR,                 -- Factory-specific webhook receiver URL
  fc_webhook_secret VARCHAR,              -- HMAC secret for webhook verification
  fc_soap_endpoint VARCHAR,               -- SOAP WSDL if factory needs SOAP
  fc_bridge_agent_config JSONB,          -- Bridge agent installer config (Tally/Busy/Marg)
  fc_tls_cert_fingerprint VARCHAR,       -- FC's TLS cert fingerprint for pinning
  fc_mtls_cert JSONB,                    -- FC's client cert for mTLS (if required)
  fc_api_spec_version VARCHAR DEFAULT '1.0', -- API spec version provisioned

  -- BUYER SIDE CONFIG (what buyer provided)
  buyer_system_type VARCHAR NOT NULL,     -- EDI_X12, ARIBA, COUPA, EDIFACT, REST, SOAP
  buyer_protocol VARCHAR NOT NULL,        -- AS2, HTTPS, SFTP, FTP, OFTP2
  buyer_endpoint_config JSONB NOT NULL,  -- {url, as2_id, sftp_host, port, paths}
  buyer_credentials JSONB,               -- ENCRYPTED: {as2_cert, api_key, oauth, shared_secret}
  buyer_edi_config JSONB,                -- {isa_qualifier, isa_id, gs_codes, version, test_ids}
  buyer_ariba_config JSONB,              -- {anid, supplier_anid, cxml_secret, commerce_url}
  buyer_coupa_config JSONB,              -- {instance_url, supplier_id, api_key}
  buyer_supported_messages JSONB,        -- ["850","855","856","810","997"] or ["OrderRequest","ShipNotice"]
  buyer_impl_guide_ref VARCHAR,          -- MinIO path to buyer's implementation guide PDF
  buyer_sample_po_ref VARCHAR,           -- MinIO path to sample PO from buyer
  buyer_api_version VARCHAR,             -- Buyer's API version
  buyer_rate_limits JSONB,               -- {requests_per_minute, daily_limit}

  -- AI MAPPING CONFIG (auto-generated, human-reviewed)
  source_mapping JSONB,                  -- JSONata: factory ERP → canonical
  target_mapping JSONB,                  -- JSONata: canonical → buyer format
  field_transforms JSONB,                -- Custom transform functions
  default_values JSONB,                  -- Defaults for fields factory doesn't have
  validation_rules JSONB,                -- Per-field validation schema
  ai_confidence_score DECIMAL(5,2),      -- AI mapping confidence 0-100%
  ai_mapping_review_status VARCHAR       -- PENDING_REVIEW, APPROVED, NEEDS_MANUAL
    DEFAULT 'PENDING_REVIEW'
    CHECK (ai_mapping_review_status IN ('PENDING_REVIEW','APPROVED','NEEDS_MANUAL')),

  -- SECURITY & VERSIONING
  security_tier VARCHAR DEFAULT 'STANDARD'
    CHECK (security_tier IN ('BASIC','STANDARD','ENTERPRISE')),  -- BASIC=API key, STANDARD=OAuth, ENTERPRISE=mTLS
  config_version INTEGER DEFAULT 1,
  previous_config_id UUID,               -- Link to previous version for rollback

  -- ONBOARDING STATE
  onboarding_status VARCHAR DEFAULT 'FACTORY_SETUP'
    CHECK (onboarding_status IN (
      'FACTORY_SETUP',        -- Factory providing ERP config + samples
      'BUYER_SETUP',          -- Awaiting buyer credentials/endpoints
      'AI_MAPPING',           -- AI generating field mappings
      'MAPPING_REVIEW',       -- Human reviewing AI-generated mappings
      'TESTING',              -- End-to-end testing in sandbox
      'CERTIFICATION',        -- Buyer certification testing (EDI)
      'READY',                -- Configuration complete, pending go-live
      'LIVE',                 -- Production active
      'SUSPENDED'             -- Temporarily suspended
    )),

  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW(),
  go_live_at TIMESTAMPTZ
);

-- Protocol support registry: what protocols FC supports per version
CREATE TABLE protocol_registry (
  protocol_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  protocol_type VARCHAR NOT NULL,        -- REST, SOAP, EDI_X12, EDIFACT, ARIBA_CXML, AS2, SFTP, OFTP2
  version VARCHAR NOT NULL,              -- "1.0", "2.0", "4010", "5010", "D96A"
  supported BOOLEAN DEFAULT TRUE,
  spec_schema JSONB,                     -- JSON Schema / XSD for this protocol version
  sample_messages JSONB,                 -- Reference sample messages
  deprecated_at TIMESTAMPTZ,
  replacement_version VARCHAR,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Onboarding audit trail
CREATE TABLE onboarding_events (
  event_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  config_id UUID NOT NULL REFERENCES connection_configs(config_id),
  event_type VARCHAR NOT NULL,           -- CONFIG_SUBMITTED, AI_MAPPING_GENERATED, TEST_PASSED, CERT_PASSED, GO_LIVE
  actor VARCHAR NOT NULL,                -- 'factory:user@acme.com', 'fc:system', 'fc:admin@fc.io', 'buyer:edi@walmart.com'
  details JSONB,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

4. AI-Powered Auto-Mapping Flow

This is the intelligence layer that makes onboarding fast. When a factory uploads sample data and a buyer PO, the AI engine performs automatic source/target detection and field mapping.

STEP 1: SOURCE DETECTION Factory uploads sample data (invoice, SO, product list) → AI identifies: format (XML/JSON/CSV/Excel) → AI identifies: ERP type (Tally XML Server response, Zoho webhook JSON, SAP B1 REST) → AI extracts: field names, data types, nesting structure → AI generates: source_mapping (ERP fields → Canonical) → Confidence score: 85-99% for known ERPs, 60-80% for custom STEP 2: TARGET DETECTION Factory uploads sample buyer PO (received from buyer) → AI identifies: buyer system type - Starts with "ISA*" → EDI X12. Extract version from GS08. - Contains "<cXML>" or "<OrderRequest>" → Ariba cXML - JSON with "purchase-order-lines" → Coupa REST - Starts with "UNB+" → EDIFACT. Extract version from UNH. - Other XML/JSON → Custom REST/SOAP. Analyze schema. → AI extracts: buyer identifiers (ISA IDs, ANID, Coupa supplier ID) → AI generates: target_mapping (Canonical → Buyer format) → AI identifies: required acknowledgments (997, ConfirmationRequest) STEP 3: FIELD MATCHING AI matches source fields to target fields through canonical model: Factory "STOCKITEM.NAME" → Canonical "product.name" → EDI "PO1*PID*F*93*" Factory "VOUCHER.AMOUNT" → Canonical "invoice.amount" → EDI "TDS*total*" Factory "GODOWN.CLOSING" → Canonical "inventory.qty" → Ariba "<SupplyQuantity>" For unmatched fields: → AI suggests: closest match with explanation → AI flags: "NEEDS_MANUAL" for ambiguous fields → AI sets: default values for buyer-required fields factory doesn't have STEP 4: VALIDATION AI generates validation rules per connection: → Required fields check (buyer-mandated) → Format validation (date formats, number precision) → Value range checks (quantities, amounts) → Cross-field validation (ship date after order date) → Buyer-specific rules (Walmart requires ASN within 24hrs of shipment)
AI Prompt Template for Mapping (Claude API call):
"Given this factory ERP sample [attached], identify all fields and map them to the FactoryConnect canonical model. For each field, provide: source_path, canonical_path, transform_function (if needed), confidence_score (0-1). Flag any fields that need human review. Also identify the ERP type and version from the data structure."

AI Prompt Template for Buyer Detection:
"Analyze this buyer purchase order [attached]. Identify: (1) system type (EDI X12/Ariba cXML/Coupa/EDIFACT/Custom), (2) version, (3) all buyer identifiers, (4) required response documents, (5) map all buyer fields to FactoryConnect canonical model with confidence scores."

5. Complete Onboarding Workflow (10 Steps)

StepPhaseActorActionsOutputDuration
1 Factory Registration Factory Signs up on FC portal. Provides: company name, GSTIN, IEC, ERP type, admin contact. WhatsApp or web form. Factory tenant created. API key generated. 10 minutes
2 ERP Connection Factory + FC Tally: Download + install Bridge Agent. Zoho: OAuth authorize. SAP B1: Provide Service Layer URL + creds. Custom REST/SOAP: Provide endpoint + auth + API version. ERP connectivity verified. First data sync. 30 min (Tally) / 5 min (cloud ERP)
3 Sample Data Upload Factory Upload: sample SO, invoice, product list, delivery note. Upload: sample buyer PO (email/PDF/EDI file). Via portal, WhatsApp photo, or email. Sample data stored in MinIO. Ready for AI. 15 minutes
4 AI Source Mapping AI Engine Claude analyzes factory ERP samples. Auto-generates source mapping (ERP → Canonical). Identifies custom fields. Sets confidence scores. source_mapping JSONB with confidence scores. 2-5 minutes (automated)
5 AI Target Detection AI Engine Claude analyzes buyer PO sample. Detects buyer system type + version. Extracts buyer identifiers. Generates target mapping (Canonical → Buyer). target_mapping JSONB. Buyer system auto-identified. 2-5 minutes (automated)
6 Buyer Credential Collection Buyer + FC FC contacts buyer IT (or factory provides contact). Collects: AS2 certs, Ariba ANID, Coupa API key, EDI IDs, SFTP creds. FC provides FC's AS2 cert + endpoints. Buyer config populated. Mutual certs exchanged. 1-5 days (buyer response dependent)
7 Mapping Review FC Admin + Factory FC admin reviews AI mappings. Factory confirms field meanings for low-confidence matches. FC finalizes default values + transform functions. Mapping status → APPROVED. 1-2 hours
8 Sandbox Testing FC + Factory End-to-end test in sandbox: Factory sends test order via ERP → FC transforms → generates buyer-format output. Factory verifies output matches expected format. Test report. All message types validated. 1-2 days
9 Buyer Certification FC + Buyer For EDI: send test transactions to buyer's test environment. Buyer validates and sends 997 FA. For Ariba: test cXML exchange. For Coupa: test API calls. Buyer certification passed. Production credentials issued. 2-10 days (buyer dependent)
10 Go-Live FC Switch connection status to LIVE. Enable production routing. Monitor first 10 transactions. Set up alerts for factory + FC ops. Connection LIVE. First real transaction flows. 1 day
Total onboarding time: 5-15 business days depending on buyer response time. Steps 1-5 (factory side + AI) complete in <1 day. Step 6 (buyer credentials) is typically the bottleneck. Target: same-day for factory side, <1 week for buyer side when factory already has buyer contact.

8. What This Enables: N Factories × M Buyers

THE N×M VISION: Factory A (Tally Prime) ──┐ Factory B (Zoho Books) ──┤ ┌── Buyer 1 (EDI X12 / AS2) Factory C (SAP B1) ──┤ ├── Buyer 2 (SAP Ariba / cXML) Factory D (Busy) ──┼── FC Canonical ────┼── Buyer 3 (Coupa / REST) Factory E (Marg) ──┤ Model Hub ├── Buyer 4 (EDIFACT / OFTP2) Factory F (ERPNext) ──┤ ├── Buyer 5 (Custom REST) Factory G (WhatsApp) ──┤ └── Buyer 6 (Custom SOAP) Factory H (CSV Upload) ──┘ Adding a new Factory-Buyer pair = 1 database record (connection_configs) AI auto-generates the mapping in minutes, not weeks No code changes. No deployments. Pure configuration. SCALE MATH: 100 factories × 3 buyers each = 300 connections 500 factories × 4 buyers each = 2,000 connections All running through the SAME canonical model All managed via the SAME onboarding workflow All mapped by the SAME AI engine
This is the network effect: Every new factory added benefits from existing buyer connections (already configured). Every new buyer added benefits from existing factory connections. Connection #1,000 is configured in hours, not weeks, because the AI has learned from the previous 999 mappings.
