Q1: How do platforms like SPS Commerce manage 10,000+ buyer-specific EDI variations?

Addresses: R-T02 (EDI Rejection Cascade), I-T04 (Buyer EDI Variations)

How SPS Commerce Does It

SPS Commerce uses a One-to-Many Canonical Mapping Model. Each supplier's internal data maps to a single SPS canonical format, then SPS maintains individual per-buyer mapping profiles that transform canonical → buyer-specific EDI. They manage 9,000+ map changes per year across their customer base. Their competitive moat is literally the library of buyer-specific maps accumulated over 20+ years.

The Architecture Pattern

Factory ERP Data
      ↓
[Source Adapter]  — Tally XML / Zoho REST / SAP OData
      ↓
FC Canonical Model  — 6 entities (Product, Order, Invoice...)
      ↓
[Buyer Profile Engine]  — Three-Layer Mapping
      │
      ├─ Layer 1: Base EDI Spec (X12 4010 standard segments)
      ├─ Layer 2: Buyer Template (Walmart-specific rules)
      └─ Layer 3: Connection Override (this specific factory→Walmart)
      ↓
[Target Adapter]  — EDI X12 / cXML / EDIFACT
      ↓
Buyer Endpoint

FC's Three-Layer Architecture (Already Designed) Is Correct

FC's existing Three-Layer Mapping architecture matches what SPS uses internally. The key execution detail is how buyer templates are populated and maintained:

Recommended Implementation

1. Buyer Template Database Schema

CREATE TABLE buyer_templates (
  id              UUID PRIMARY KEY,
  buyer_name      TEXT NOT NULL,           -- "Walmart", "Target", "Costco"
  edi_version     TEXT NOT NULL,           -- "4010", "5010"
  transaction_set TEXT NOT NULL,           -- "850", "856", "810"
  template_rules  JSONB NOT NULL,          -- segment requirements, qualifiers, code values
  validation_rules JSONB NOT NULL,         -- buyer-specific validation (zero-tolerance fields)
  last_verified   TIMESTAMP,              -- when we last verified against buyer's IG
  ig_version      TEXT,                    -- buyer's Implementation Guide version
  created_at      TIMESTAMP DEFAULT NOW()
);

-- Example: Walmart 850 template
-- template_rules contains:
-- { "required_segments": ["ISA","GS","ST","BEG","DTM","N1","PO1","CTT","SE"],
--   "beg_purpose_code": "00",
--   "dtm_qualifier": "002",
--   "n1_qualifiers": ["ST","BT","VN"],
--   "po1_unit_codes": ["EA","CA","PK"],
--   "cumulative_qty": true }

2. Pre-Flight Validation Engine

Before any EDI document is sent to a buyer, it passes through a 3-stage validator:

  • Stage 1 — Structural: Segment counts, delimiters, envelope integrity (ISA/GS/ST match)
  • Stage 2 — Semantic: Code values exist in X12 codelist, dates are valid, quantities are positive
  • Stage 3 — Buyer-Specific: All required segments per buyer template present, qualifiers match buyer's IG, conditional rules satisfied

3. Template Acquisition Strategy

  • Phase 1 (Manual): Obtain Implementation Guides (IGs) from 10 target buyers. Manually extract rules into template_rules JSONB. Budget 2-3 days per buyer template.
  • Phase 2 (AI-Assisted): Build IG Parser — Claude/GPT reads buyer IG PDF, extracts segment requirements, generates template_rules JSON. Human verifies. Reduces to 4-6 hours per template.
  • Phase 3 (Community): Templates contributed by factories that pass certification. Verified, versioned, shared across FC customers connecting to same buyer.
Verdict: FC's Three-Layer Mapping is architecturally correct. Execution priority: Acquire 10 buyer IGs immediately (Walmart, Amazon, Target, Costco, Home Depot, Kroger, Macy's, JCPenney, Best Buy, Lowe's). Build template extraction pipeline. This is the moat — whoever has the most accurate buyer templates wins.

Recommended Path: Dual Strategy

Phase 1: AWS Transfer Family (Quick Start)

Use AWS Transfer Family for first 10-20 buyer AS2 connections. Managed service = fewer ops headaches while team is small. Accept the limitations. Budget ~$700-1000/month.

Phase 2: OpenAS2 Self-Hosted (Scale)

When buyer connections exceed 30+ or AWS limitations become blocking, deploy OpenAS2 on GKE as a Kubernetes deployment. Containerize with health checks. Cost drops to ~$100/month. Full control over certificate management, message flow, and protocol extensions.

Phase 3: SFTP Fallback for All Buyers

Configure SFTP delivery as secondary path for every buyer that supports it. When AS2 fails, automatic failover to SFTP. Most large buyers accept both AS2 and SFTP.

Verdict: Start with AWS Transfer Family for simplicity. Budget migration to self-hosted OpenAS2 at 30+ connections. The $216/month server cost becomes the floor, not the ceiling, as you scale. Maintain SFTP as universal fallback.

The 5-Step Certification Process

1. Kickoff/Mapping (2-4 weeks): Obtain buyer's Implementation Guide. Map data fields. Build buyer template.
2. Unit Testing (1-2 weeks): Internal validation. Generate test documents. Validate against buyer template rules.
3. Integration Testing (2-4 weeks): Exchange test data with buyer's testing environment. Fix issues. Resubmit.
4. Certification Testing (1-2 weeks): Formal compliance validation by buyer's EDI team.
5. Cutover (1 week): Production go-live. Shadow mode first, then live.

FC Certification Acceleration Strategy

  • Pre-Built Test Harness: For each target buyer, build a mock buyer endpoint that replicates their exact validation rules. Run 100% of test cases internally before submitting to buyer. Reduces buyer-side rounds from 3 to 1-2.
  • Parallel Certification: Certify with 3-5 buyers simultaneously. Don't serialize. Each buyer certification is independent.
  • Shadow Mode (2 weeks): Before formal testing, run FC in shadow mode with factory's existing EDI. Compare FC output to existing output. Fix discrepancies before entering buyer's test cycle.
  • Certification Tracker: Build a dashboard tracking certification status per buyer: test cases submitted, pass/fail, open issues, next action, due date.
Verdict: Budget 8-12 weeks per buyer for first certification. After 5+ certifications, internal process matures and timeline drops to 4-6 weeks. The critical investment is building accurate buyer test harnesses — this is what SPS Commerce has spent 20 years accumulating.

Three-Phase Buyer Acceptance Strategy

Phase 1: VAN Partnership Bridge (Month 1-6)

Partner with an established VAN (TrueCommerce or Cleo) as a technology partner. FC generates the EDI, sends to VAN's infrastructure, VAN delivers to buyer. Buyer sees TrueCommerce/Cleo — not FC. FC is invisible to buyer initially.

Part 3: Cloud Architecture Expert Answers

Q11: GKE Autopilot + KEDA at Scale

Addresses: R-T09 (Kubernetes Complexity), A-T06 (GKE Autopilot Handles Pattern)

Research Findings

KEDA on GKE Autopilot is production-ready and officially documented by Google Cloud.

  • Google Cloud has published official tutorials for KEDA scale-to-zero on Autopilot
  • KEDA scales Pub/Sub, Redis, Kafka, and custom metrics workloads from 0→N pods
  • Cold start latency: 5-30 seconds per pod when scaling from zero (this is the main trade-off)
  • No published per-cluster deployment limit, but 50-200 KEDA scalers per cluster is the practical range before controller becomes bottleneck
  • ~60-70% cost reduction vs. maintaining minimum 2 replicas for bursty workloads

FC Deployment Strategy

Scale-to-Zero (KEDA) — Bursty/Low-Traffic Adapters

Adapters with intermittent traffic: tgt-coupa-31, tgt-edifact-d01b, src-marg-api-v9, src-busy-odbc-21. These may handle 0-100 messages/day. Scale to zero between bursts. Accept 5-30 second cold start.

Always-On (HPA) — Critical Path Adapters

Adapters on the critical path: tgt-x12-4010, src-tally-json-70, src-zoho-v3. These handle continuous traffic. Min 2 replicas. HPA scales up based on CPU/queue depth.

Always-On (Fixed) — Core Services

Routing engine, canonical transform, pre-flight validator, cascade dependency engine. These are in every message's path. Fixed 2-3 replicas. No scale-to-zero.

Verdict: GKE Autopilot + KEDA is the right choice. 25 deployments is well within the supported range. The key decision: don't scale-to-zero on critical-path services (routing, validation). Only scale-to-zero on per-buyer/per-ERP adapters with intermittent traffic.

Recommended Path

Phase 1: Redis Streams (You Already Have Redis)

FC already uses Redis for caching. Adding Streams for message routing is zero additional infrastructure. Redis Sentinel (3 nodes) for HA. AOF with fsync=everysec. Dual-write message log to PostgreSQL for durability guarantee. This handles FC's Phase 1 volume (estimated 1K-10K msg/min) with massive headroom.

Phase 2 (If Volume Exceeds 50K msg/min): NATS JetStream

Single binary deployment. Built-in clustering. No external coordinator. Stronger persistence than Redis. Better consumer group semantics. Perfect for a small team that doesn't want Kafka's operational burden.

Phase 3 (If Volume Exceeds 300K msg/min): Kafka

Kafka 4.0+ dropped ZooKeeper (KRaft mode). Still operationally heavy. Only needed if FC grows to thousands of concurrent connections with high message volume. Unlikely before Phase 3.
