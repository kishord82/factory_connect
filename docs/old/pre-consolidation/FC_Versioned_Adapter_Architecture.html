FactoryConnect — Versioned Adapter Architecture & Network Security

Independent, load-scalable microservice adapters for every source/target version with secure network layer, cipher management, and zero-downtime version lifecycle | April 2026

1. Design Philosophy: Think Big, Keep It Simple

Kishor's principle: "Think big and solve with simple — not too complex to build." Every decision in this architecture follows three rules:

1
One Adapter = One Container
Each source/target version is a single Docker container with one job. No shared state, no tangled dependencies.
2
Scale by Load, Not by Hope
Kubernetes HPA scales each adapter independently based on actual message volume. Busy adapters get more pods; idle ones scale to zero.
3
Deploy & Decommission Freely
Add a new version without touching existing ones. Sunset an old version when zero connections remain. No "big bang" migrations.
The Core Insight: FactoryConnect connects N factories (each with different ERP types and versions) to M buyers (each with different procurement systems and versions). Instead of building a monolithic translator that handles every combination, we build small, focused adapters — one per source version, one per target version — and let the Canonical Model in the middle be the universal translator. The routing engine picks the right adapter pair for each message. That's the entire complexity — everything else is just plumbing.

How It Works — 5 Steps, That's It

  1. Message arrives from factory (webhook, poll, bridge push) into the matching Source Adapter
  2. Source Adapter transforms raw ERP data into FactoryConnect Canonical Model (6 entities: Product, Inventory, Order, Shipment, Invoice, Party)
  3. Routing Engine looks up connections + routing_rules tables to find which buyer(s) should receive this, and which Target Adapter version to use
  4. Target Adapter transforms Canonical Model into buyer-specific format (EDI 850, cXML OrderRequest, Coupa PO, etc.) using the three-layer mapping (Base Spec + Buyer Template + Connection Overrides)
  5. Target Adapter delivers over the correct protocol (AS2, HTTPS, SFTP, cXML, SOAP) with proper security credentials

5. Load-Based Auto-Scaling (Every Adapter Independently)

This is where Kishor's insight about load-based scalability becomes architecture. Each adapter is a Kubernetes Deployment with its own Horizontal Pod Autoscaler (HPA). The adapters that handle heavy traffic get more pods; adapters for rarely-used ERP versions scale to zero when idle.

Why This is Simple: We don't build a custom load balancer. Kubernetes does it for us. We just tell it: "This adapter should have 1-20 pods, and scale up when the message queue depth exceeds 50." That's 15 lines of YAML per adapter. Zero custom code.

kind: HorizontalPodAutoscaler

  name: tgt-x12-5010-hpa
  namespace: fc-adapters
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: tgt-x12-5010
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: External
    external:
      metric:
        name: redis_queue_depth
        selector:
          matchLabels:
            queue: "outbound.x12.5010"
      target:
        type: AverageValue
        averageValue: "50"     # Scale up when > 50 msgs per pod
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 30   # React fast to load spikes
      policies:
      - type: Pods
        value: 4                       # Add up to 4 pods at a time
        periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300  # Wait 5 min before scaling down
      policies:
      - type: Pods
        value: 2
        periodSeconds: 120

Scale-to-Zero Economics

At 500 factories in Year 1, not every adapter runs 24/7. With KEDA, adapters for rarely-used systems (SAP B1 source, Coupa target, EDIFACT target, Generic REST/SOAP) sleep until work arrives. This cuts infrastructure cost by an estimated 30-40% vs running all adapters at minimum 1 pod.

Always Running (24/7)

  • src-tally-3, src-tally-4 (most factories)
  • src-zoho-v2 (webhook listener)
  • src-whatsapp-v1 (always online)
  • tgt-x12-5010, tgt-x12-4010 (bulk of US buyers)
  • tgt-ariba-23 (major buyer system)
  • Routing Engine + Canonical Model Engine

Scale-to-Zero (Wake on Demand)

  • src-sapb1-10 (fewer SMEs on SAP B1)
  • src-csv-v1 (batch uploads only)
  • tgt-coupa-31 (growing but lower volume)
  • tgt-edifact-d01b (EU buyers, CBAM growth)
  • tgt-oracle-v2 (selective buyers)
  • All generic adapters (REST, SOAP)

The "Two in Production" Pattern: When a new version arrives (e.g., Tally 4.x releases a breaking API change for Tally 5.x), we deploy src-tally-5 alongside src-tally-4. Both run simultaneously. Existing Tally 4.x factories keep working on src-tally-4. New factories on Tally 5.x get src-tally-5. Over time, as factories upgrade, traffic shifts naturally. When src-tally-4 reaches zero connections, it gets decommissioned. No forced migrations. No downtime.

Automatic Decommission Logic

// Runs daily as a cron job in the routing engine
func checkAdapterSunset(ctx context.Context) {
    deprecated := getAdaptersByStatus("DEPRECATED")
    for _, adapter := range deprecated {
        activeConns := countActiveConnections(adapter.AdapterID)
        if activeConns == 0 {
            daysSinceZero := daysSinceLastConnection(adapter.AdapterID)
            if daysSinceZero >= 30 {
                // Safe to remove — 30 days with zero traffic
                scaleToZero(adapter.AdapterID)
                markSunset(adapter.AdapterID)
                archiveContainerImage(adapter.ContainerImage)
                notifyOps(fmt.Sprintf("Adapter %s sunset after 30d zero traffic", adapter.AdapterID))
            }
        }
    }
}

7. Routing Engine: Picking the Right Adapter Pair

The Routing Engine is the brain. When a message arrives, it determines which source adapter processed it, which buyer(s) it goes to, and which target adapter version to use for each buyer. It uses the connection_adapter_bindings table for fast lookup.

// Core routing logic — simple and fast
func routeMessage(ctx context.Context, msg CanonicalMessage) error {
    // 1. Find all buyer connections for this factory
    connections := getActiveConnections(msg.FactoryID)

    for _, conn := range connections {
        // 2. Look up bound adapters for this connection
        binding := getAdapterBinding(conn.ConnectionID)

        // 3. Verify target adapter is still ACTIVE or DEPRECATED (not SUNSET)
        adapter := getAdapter(binding.TargetAdapterID)
        if adapter.LifecycleStatus == "SUNSET" {
            // This shouldn't happen, but safety check
            alertOps("Routing to sunset adapter", binding.TargetAdapterID)
            continue
        }

        // 4. Load three-layer mapping for this connection
        mapping := loadMapping(conn.ConnectionID)

        // 5. Enqueue to the target adapter's work queue
        queueName := fmt.Sprintf("outbound.%s", binding.TargetAdapterID)
        enqueue(queueName, TransformJob{
            Message:      msg,
            ConnectionID: conn.ConnectionID,
            Mapping:      mapping,
            IdempotencyKey: msg.IdempotencyKey,
        })
    }
    return nil
}
Why Queue-Based, Not Direct Call: Source adapters don't call target adapters directly. They produce a Canonical Model message, and the Routing Engine enqueues it to the correct target adapter's Redis queue. This decouples source and target completely — a target adapter can be restarted, scaled, or replaced without affecting sources. And it gives us natural backpressure: if a target adapter falls behind, the queue fills, KEDA scales it up.

8. Version Flexibility: Handling Any Versioning Problem

Kishor said: "We should be flexible to any kind of versioning problems." Here's how the architecture handles every versioning scenario we've identified:

Versioning ProblemHow the Architecture Handles ItComplexity
ERP releases a new version with breaking API changes
e.g., Tally Prime 4.x → 5.x changes XML schema		 Deploy a new source adapter (src-tally-5) alongside the existing one. Both run. No migration needed. Route new factories to the new adapter. Low New adapter, ~2 days
Buyer changes EDI version requirement
e.g., Walmart moves from X12 4010 → 5010		 Update the buyer's connection_adapter_bindings to point to tgt-x12-5010. Existing mapping layers carry over (buyer template + connection overrides). Test in sandbox, flip the binding. Low Config change + test
Same buyer, different versions for different document types
e.g., PO on 5010 but Invoice on 4010		 Connection bindings are per document type, not per connection. One factory-buyer pair can have tgt-x12-5010 for POs and tgt-x12-4010 for invoices. Low Routing rule config
Minor version patch with field changes
e.g., Zoho Books API v2.3 adds a new field		 Schema fingerprinting detects the drift. If it's additive (new fields only), AI auto-heals the mapping. If it's breaking, pause and alert. No new adapter needed for minor changes — the existing adapter handles it via config update. Low Auto-heal or config
Protocol change (not just version)
e.g., Buyer switches from AS2 to SFTP, or from SOAP to REST		 The protocol is part of the target adapter. If buyer moves from SOAP to REST, create a new binding to tgt-generic-rest instead of tgt-generic-soap. Or build a new adapter if it's a major buyer. Medium New binding or adapter
Completely new ERP system enters the market
e.g., A new Indian ERP startup gains traction		 Build a new source adapter from the template. Plug into the adapter mesh. Register in adapter_versions. Done. No changes to routing engine, canonical model, or any target adapter. Medium 1-2 weeks for new adapter
Regulatory format change
e.g., EU CBAM report format v2 replaces v1		 Build tgt-cbam-v2 alongside tgt-cbam-v1. Route based on effective date. Keep both until the deadline passes. Low Standard lifecycle
Factory uses multiple ERP systems simultaneously
e.g., Tally for accounting, Zoho for inventory		 Each ERP is a separate source connection to the same factory. Routing engine merges data by factory ID in the canonical model. Two source adapters, same factory. Low Multi-source config
The Universal Answer: For any versioning problem, the solution is always one of: (a) deploy a new adapter beside the old one, (b) update a config/mapping, or (c) update a connection binding. The core engine never changes. That's the power of the versioned microservice pattern — the complexity is isolated in small, replaceable containers.

10. Cipher Suite Allowlist

FactoryConnect enforces a strict cipher allowlist. Only modern AEAD (Authenticated Encryption with Associated Data) ciphers are permitted. Legacy ciphers (RC4, 3DES, CBC mode) are rejected.

AS2 Note: Many older AS2 implementations use 3DES for S/MIME message encryption. FC will NOT support 3DES even for AS2. If a buyer's AS2 system only supports 3DES, FC will flag this during onboarding certification and require the buyer to upgrade their AS2 software to support AES-128-CBC or AES-256-CBC for S/MIME (this is separate from the TLS cipher — S/MIME encrypts the payload inside the TLS tunnel). This is a firm security boundary.

        CipherSuites: nil, // For TLS 1.3, Go uses the built-in list (cannot be customized)

        // For TLS 1.2 fallback (legacy AS2 buyers only, controlled by feature flag)
        // CipherSuites: []uint16{
        //     tls.TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384,
        //     tls.TLS_ECDHE_RSA_WITH_CHACHA20_POLY1305_SHA256,
        // },

        CurvePreferences: []tls.CurveID{
            tls.X25519,    // Preferred: fastest, most secure
            tls.CurveP256, // Fallback: widely supported
        },

        // Disable session tickets for forward secrecy
        SessionTicketsDisabled: true,
    }

    // mTLS: require client certificate for factory bridge connections
    if requireClientCert {
        caCert, _ := os.ReadFile(secrets.MustGet("FC_CA_CERT_PATH"))
        caCertPool := x509.NewCertPool()
        caCertPool.AppendCertsFromPEM(caCert)
        cfg.ClientCAs = caCertPool
        cfg.ClientAuth = tls.RequireAndVerifyClientCert
    }

    return cfg
}

11. Per-Connection Security Configuration

Each connection (factory → FC → buyer) has its own security configuration stored in the database. This allows FC to support heterogeneous security requirements across hundreds of connections without any hardcoded assumptions.

CREATE TABLE connection_security (
    connection_id       UUID PRIMARY KEY REFERENCES connections(id),

    -- Inbound (factory → FC)
    inbound_tls_min     VARCHAR(10) DEFAULT 'TLS_1_3',
    inbound_auth_type   VARCHAR(20) NOT NULL,
        -- 'mtls', 'api_key_jwt', 'webhook_hmac', 'basic_auth'
    inbound_client_cert VARCHAR(200),            -- Vault path to factory's client cert
    inbound_api_key_id  VARCHAR(100),            -- Reference to API key in Vault
    inbound_ip_allowlist INET[],                 -- Optional IP restrictions

    -- Outbound (FC → buyer)
    outbound_protocol   VARCHAR(10) NOT NULL,    -- 'AS2', 'HTTPS', 'SFTP', 'OFTP2', 'SOAP'
    outbound_tls_min    VARCHAR(10) DEFAULT 'TLS_1_3',
    outbound_tls_legacy BOOLEAN DEFAULT FALSE,   -- Allow TLS 1.2 for this buyer?
    outbound_cipher_policy VARCHAR(20) DEFAULT 'STRICT',
        -- 'STRICT' = TLS 1.3 only, 'COMPAT' = TLS 1.2 ECDHE-GCM allowed

    -- AS2-specific
    as2_signing_alg     VARCHAR(20) DEFAULT 'SHA-256',
    as2_encrypt_alg     VARCHAR(20) DEFAULT 'AES-256-CBC',
    as2_mdn_type        VARCHAR(10) DEFAULT 'SYNC',  -- SYNC or ASYNC
    as2_buyer_cert_path VARCHAR(200),             -- Vault path to buyer's public cert
    as2_fc_key_path     VARCHAR(200),             -- Vault path to FC's signing key

    -- OAuth (Coupa, Oracle)
    oauth_token_url     VARCHAR(500),
    oauth_client_id     VARCHAR(200),             -- Vault path
    oauth_client_secret VARCHAR(200),             -- Vault path
    oauth_scope         VARCHAR(200),

    -- SOAP WS-Security
    ws_security_type    VARCHAR(20),              -- 'UsernameToken', 'X509Token', 'SAML'
    ws_security_cred    VARCHAR(200),             -- Vault path

    -- Certificate lifecycle
    cert_expires_at     TIMESTAMPTZ,
    cert_auto_renew     BOOLEAN DEFAULT TRUE,
    cert_renewed_at     TIMESTAMPTZ,

    created_at          TIMESTAMPTZ DEFAULT NOW(),
    updated_at          TIMESTAMPTZ DEFAULT NOW()
);

-- Alert on expiring certificates (daily cron check)
CREATE INDEX idx_cert_expiry ON connection_security(cert_expires_at)
    WHERE cert_expires_at IS NOT NULL;

Kubernetes HPA + KEDAAuto-scale on queue depth15 lines of YAML per adapter, zero custom scaling codeEach adapter scales independently based on actual load

17. Recommendations & Next Steps

  1. Build the Adapter Template First — Create a cookiecutter/template repo for adapters. Every new adapter starts from this. Includes: main.go, transform.go, Dockerfile, k8s manifests, test harness, Prometheus metrics. Target: 2 hours to spin up a new adapter skeleton.
  2. Start with 4 Adapters — Phase 1: src-tally-3, src-tally-4, tgt-x12-5010, tgt-ariba-23. These cover 80% of initial factory-buyer combinations.
  3. Implement Routing Engine as a Separate Service — Stateless Go service that reads from connections + connection_adapter_bindings. Scales independently from adapters.
  4. Set Up KEDA Early — Install KEDA on GKE from Day 1. Even if all adapters start at min 1, the ScaledObjects are ready for when traffic patterns emerge.
  5. Certificate Management Automation — Use cert-manager with Let's Encrypt for FC-side certificates. Vault PKI for internal mTLS. Automate buyer certificate ingestion during onboarding.
  6. Golden Test Suite per Adapter — Each adapter must have 20+ real-world test cases before going ACTIVE. These become the contract tests that prevent regressions.
  7. Security Scanning in CI — Trivy for container image scanning, gosec for Go code scanning. Block deployment if critical CVEs found.
  8. Build the Generic Adapters (REST + SOAP) Last — These are the most complex because they're configurable. Get specific adapters working first, then extract the generic pattern.
